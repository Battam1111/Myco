# MCP 2026 Roadmap — Implications for Myco

## Overview (Roadmap updated March 5, 2026)

The Model Context Protocol roadmap for 2026 is organized around four priority areas with Working Groups driving delivery. This note captures what matters for Myco as an MCP server.

## Priority Area 1: Transport Evolution and Scalability
- **Streamable HTTP evolution**: servers must scale horizontally without holding state; sessions must survive server restarts and scale-out
- **Scalable session handling**: session creation, resumption, and migration transparency
- **MCP Server Cards**: standard for .well-known URL exposing server metadata for discovery by browsers, crawlers, registries
- **No new transports this cycle**: STDIO (local) and Streamable HTTP (remote) remain the two official transports

### Myco Impact: Myco currently uses STDIO transport only. If Myco wants remote deployment (multi-user, team use), it needs to implement Streamable HTTP. Server Cards would help Myco be discoverable in MCP registries.

## Priority Area 2: Agent Communication
- **Tasks primitive (SEP-1686)**: call-now / fetch-later pattern for agents
- **Retry semantics**: what happens when a task fails transiently
- **Expiry policies**: how long results retained after completion

### Myco Impact: Myco's long-running operations (hunger execute, compress, lint) could benefit from the Tasks primitive for async execution. Currently these are synchronous tool calls.

## Priority Area 3: Governance Maturation
- Contributor Ladder: community participant -> WG contributor -> WG facilitator -> lead maintainer -> core maintainer
- Delegation model for WGs to accept SEPs independently
- Charter templates for WGs/IGs

### Myco Impact: Myco could participate in relevant Working Groups (particularly Enterprise WG and potentially a Knowledge/Memory IG) to influence the standard.

## Priority Area 4: Enterprise Readiness
- Audit trails and observability (end-to-end visibility)
- Enterprise-managed auth: SSO-integrated flows, Cross-App Access (XAA)
- Gateway and proxy patterns: behavior when client routes through intermediary
- Configuration portability: configure server once, works across MCP clients

### Myco Impact: If Myco targets enterprise adoption, it needs audit trail support (log.md is already an audit trail), auth integration, and gateway compatibility.

## On the Horizon (Not Top Priority)
- **Triggers and Event-Driven Updates**: servers proactively notify clients of state changes (webhooks). This would let Myco push hunger signals to clients instead of requiring polling.
- **Result Type Improvements**: streamed/reference-based results for large payloads. Relevant for Myco's large lint reports or search results.
- **Security**: finer-grained least-privilege scopes, DPoP (SEP-1932), Workload Identity Federation (SEP-1933)
- **Extensions Ecosystem**: Skills primitive for composed capabilities. Myco's 19 tools could potentially be composed into higher-level Skills.

## Authorization Specification (Newly Detailed)
- OAuth 2.1 as foundational framework
- MCP server = OAuth Resource Server
- Authorization Code grant as primary mechanism
- PKCE mandatory for public clients

### Myco Impact: Myco's MCP server currently has no auth. For remote deployment, OAuth 2.1 implementation is required.

## Key Takeaway
Myco is well-positioned architecturally (STDIO works, tools are well-designed) but needs work for remote/enterprise deployment: Streamable HTTP transport, OAuth 2.1 auth, Server Card metadata, and potentially async Tasks for long operations.
