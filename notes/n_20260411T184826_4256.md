## What the announcement actually is

Anthropic publicly launched **Claude Managed Agents** on 2026-04-08 — a suite of composable APIs for constructing and deploying cloud-hosted agents at scale. The offering handles sandboxing, auth, tool execution management, long-running autonomous sessions with persistent state, multi-agent delegation (research preview), and a governance framework (scoped permissions, identity management, execution tracing). Performance claim: up to ~10-point improvement on structured-file-generation tasks vs standard prompting. Entered public beta on the Claude Platform at announcement time.

## The positioning question — Myco vs Managed Agents

These are **different product categories** and should not be confused.

**Managed Agents** is a hosted-runtime product for companies that want to ship agent features without building infrastructure. It sells: sandboxing, session persistence, delegation graph, governance, observability. Target buyer: engineering orgs at named enterprises. Revenue model: hosted API usage.

**Myco** is a substrate for how an agent's durable state is organized, gated, and compressed across sessions. It sells: zero, it's a framework. Target user: individuals or small teams who want their agent's memory to be legible, auditable, and compounding. Revenue model: none, it's upstream open-source.

**Overlap surface — low, but non-zero.** Both care about "agent state persists across sessions". Managed Agents handles this by owning the session store in the cloud; Myco handles it by making `notes/*.md` the source of truth on disk. The two could literally stack: a Managed-Agents-hosted agent could write to a Myco substrate mounted via MCP, with Managed Agents providing the runtime and Myco providing the durable-state contract. This is *exactly* the same layering story as the hermes-agent pressure test — runtime on top of substrate, neither substitutes the other.

**Divergent bet.** Managed Agents bets that the winning shape is "agents as managed composable APIs" — the agent is a service endpoint you call, and the hard parts (sandbox, auth, trace) are infrastructure concerns handled by the platform. Myco bets that the winning shape is "substrate compounds across sessions because it has discipline" — the agent is fundamentally a process that eats, digests, and integrates, and the hard parts are metabolic (compression, lifecycle, lint). These are orthogonal claims. Both can be true simultaneously. **Myco should not position itself as an alternative to Managed Agents** — they play different games.

## Three concrete points for Myco's positioning doc

**1. The enterprise naming is the hard signal.** Notion, Rakuten, Asana are not experimental adopters — they are household-name orgs going on-record with named deployments. This tells Myco that by 2026-04-08 the "agent-as-service" framing has moved past pilot into production. Myco's positioning should **acknowledge that Managed Agents occupies the "enterprise agent runtime" space** and not attempt to compete there. Myco's niche is adjacent: durable, legible, individually-owned substrate.

**2. The "governance framework" in Managed Agents is the interesting cross-reference.** Scoped permissions, identity management, execution tracing — all of these are *about trust at runtime*. Myco's craft protocol and 15-dimension lint are *about trust across sessions* (does this note meet the compression discipline, does this kernel-contract change have the rounds/confidence trail). Different trust questions, same underlying concern. A future Myco positioning doc could explicitly frame it as: *Managed Agents answers "can I trust this agent *now*", Myco answers "can I trust this substrate *across time*"*.

**3. The performance claim (+10 points on structured file generation) is a data point to track.** Myco has no runtime performance claim and shouldn't make one — it's not a runtime. But when Myco eventually measures anything empirically, the Managed Agents blog is a reference for how Anthropic frames a performance claim in public: "up to X-point improvement on task Y per internal testing". That's a responsible framing template if Myco ever needs one.

## What's explicitly NOT portable and shouldn't be absorbed

**The hosted-runtime architecture.** Everything in Managed Agents that makes it a product (infra abstraction, hosted sessions, multi-tenant governance) is the *opposite* of Myco's "markdown files on local disk" commitment. Attempting to absorb architectural ideas here would invert Myco's core bet.

**The "composable APIs" framing.** Myco is not an API surface for agents — it's a file-on-disk contract with a lifecycle and a lint. The API-as-product framing doesn't match.
