# Wave 39 — L20 Translation Mirror Consistency lint (v0.30.0) — DECISIONS

**Wave**: 39
**Date landed**: 2026-04-12
**Contract**: v0.29.0 → v0.30.0 (minor — additive lint dimension)
**Class**: kernel_contract (3 rounds, current_confidence = target_confidence = 0.90)
**Craft**: `docs/primordia/translation_mirror_lint_craft_2026-04-12.md`

## Why this wave existed

Wave 26 D3 friction-driven ordering elected Wave 37 D7 followup #2
(translation-mirror lint) as the next wave after Wave 38 closed followup #1
(L19 lint dimension count consistency). The three locale READMEs were a
hidden silent-rot surface: between Wave 30 and Wave 37, structural drift
accumulated between `README.md` / `README_zh.md` / `README_ja.md` (sections
added to one and forgotten in others). L19 catches narrative count drift;
L20 catches the orthogonal structural drift class. Together they make
locale READMEs a load-bearing immune surface instead of three independently-
rotting copies.

## Decisions

- **D1**: SSoT for parity is the 5-tuple `skeleton(content) =
  (h2_count, h3_count, code_count, table_rows, badge_count)`. Tuple equality
  across all participating locale READMEs is the parity check. Rejected
  alternatives: full prose hash (too brittle, breaks on every translation
  edit), word-count comparison (too noisy, varies wildly across languages),
  AST-level markdown parse (over-engineered, no markdown AST library is in
  the dependency budget).

- **D2**: `README.md` is the reference. If missing (greenfield project),
  fallback to the locale README with the highest h2_count (most-developed).
  If fewer than 2 locale READMEs exist, L20 silently passes — there is
  nothing to mirror against. Rejected alternatives: alphabetical-first
  (arbitrary), random selection (non-deterministic), require all 3 always
  (over-strict for early projects).

- **D3**: L20 is a **permanent structural lint dimension**, not a one-shot
  audit. It runs on every `myco lint` invocation. The substrate's locale
  READMEs are a recurring drift class — every wave that touches READMEs
  (e.g. badge bumps, version-row updates, new section adds) is a fresh
  drift opportunity. A one-shot audit would be a Wave 26 anti-pattern
  ("we did it once, now it's done") instead of a regression-test guarantee.

- **D4**: Severity = HIGH for any drifted skeleton component. README is
  the user-facing first-impression surface in three languages; soft-warning
  severity (MEDIUM/LOW) would normalize ignoring the rot. The same severity
  rationale Wave 38 D4 used for L19 README badge drift applies here.

- **D5**: Skip-marker escape hatch — a section preceded by `<!-- l20-skip -->`
  on the previous non-blank line is excluded from the skeleton count. This
  is the operator's escape hatch for genuinely locale-specific sections
  (e.g. a Chinese-only "致谢" or Japanese-only "謝辞") that have no
  structural counterpart in English. Without this, operators would have
  to remove locale-native content to keep L20 green — exactly the wrong
  incentive. The marker is conservative (only the immediately following
  H2/H3 is excluded; subsequent structure resumes counting) so it cannot
  accidentally suppress more than intended.

- **D6**: Code-fence-aware parser. `_count_skeleton` is a stateful line
  walker tracking fence open/close state. H2/H3/`|`/badge lines inside
  ` ``` ` blocks are NOT counted. This guards against false positives
  from inline markdown samples (e.g. a `## fake heading` inside a
  ` ```markdown ` fence). The parser handles trailing whitespace via
  `rstrip()` and CRLF via `splitlines()` transparently.

- **D7**: Contract bump v0.29.0 → v0.30.0 in `_canon.yaml`,
  `src/myco/templates/_canon.yaml`, and `docs/contract_changelog.md`.
  `synced_contract_version` mirrors the bump to keep L17 quiescent. Minor
  bump matches the precedent set by L18 (v0.26.0), L19 (v0.29.0), and the
  Wave 24 L17 landing — additive lint dimensions never warrant major bumps.

- **D8**: 4 unit tests added at `tests/unit/test_lint_translation_mirror.py`,
  one per scar class: `test_l20_clean_substrate_passes` (D1 base case),
  `test_l20_section_drop_caught` (D1+D4 principal scar),
  `test_l20_skip_marker_respected` (D5 escape hatch),
  `test_l20_code_fence_aware` (D6 parser invariant). Suite count advances
  26 → 30. Tests pull `_count_skeleton` and `lint_translation_mirror_consistency`
  directly (no CLI roundtrip) and use the existing `_isolate_myco_project`
  fixture from `tests/conftest.py`.

- **D9**: Operational landing — bump 15+ narrative surfaces from
  20-dimension/L0-L19/Lint-20%2F20 to 21-dimension/L0-L20/Lint-21%2F21.
  L19 dogfooded the transition: every surface that still claimed
  "20-dimension" surfaced as a HIGH issue at the first lint run, forcing
  the entire change-set to land in lockstep. This proves L19 (Wave 38)
  works as designed and proves the SSoT discipline is self-enforcing
  through wave transitions. Bonus: closed a Wave 38 oversight where
  `src/myco/immune.py` was missing `lint_dimension_count_consistency` from
  its imports + `__all__`; Wave 39 fixed both that and added L20 in the
  same patch.

## Limitations (honest)

- **L1**: Skeleton parity is structural, not semantic. Two READMEs can have
  identical 5-tuples while their content drifts apart in meaning. L20
  catches "the table got removed from one locale" but not "the table got
  rewritten with different rows in one locale". The 5-tuple is a
  necessary-not-sufficient signal; semantic parity remains a human review
  concern.

- **L2**: The badge regex `\bLint-\d+%2F\d+\b` only matches the URL-encoded
  shields.io form. Non-shields.io badges (e.g. a custom SVG) won't be
  counted. Acceptable because shields.io is the de-facto convention.

- **L3**: `_count_skeleton` parses line-by-line and is not a full markdown
  parser. Edge cases like indented fences (`    \`\`\``) or HTML `<table>`
  tags are not handled. Current substrate uses neither.

- **L4**: The locale allowlist `_L20_LOCALE_READMES = ("README.md",
  "README_zh.md", "README_ja.md")` is hardcoded. Adding `README_es.md`
  requires a code change. Intentional (Wave 39 D1 §C7) to avoid drive-by
  detection over-firing on adapter-internal locale variants, but means
  new locales need a one-line patch.

## Wave 40+ disposition

The remaining Wave 37 D7 followup is **#3 — contract-version-inline lint**.
This will catch the orthogonal scar class where:
- `docs/contract_changelog.md` claims a version disagreeing with `_canon.yaml::system.contract_version`
- `__version__` strings in Python modules drift away from canonical contract version
- Inline doc references (`v0.x.y` mentions in MYCO.md, README, etc.) fall behind canon

Wave 40 will be the kernel_contract craft + impl for this — same shape as
Waves 38 + 39 (3 rounds, kernel_contract class, contract bump, ~4 unit tests,
narrative-surface bump driven by L19's existing dogfooding mechanism).

After Wave 40 lands, the substrate's three top staleness scar classes
(dimension count, locale skeleton, version string) all become regression-
testable lint dimensions. Wave 26 D3 friction-driven ordering then re-
evaluates against fresh friction signals from the post-Wave-40 state.

## Why this matters (one paragraph)

Wave 38 made the dimension count `len(FULL_CHECKS)` a single source of truth
and gave it ground-truth lint coverage (L19). Wave 39 makes the locale
README skeletons mirror each other and gives that mirror ground-truth lint
coverage (L20). Together, the two waves close the two highest-leverage
silent-rot surfaces in the substrate's user-facing first-impression layer.
Future waves can add new dimensions, new sections, new badges — and all
three locales will move in lockstep because the immune system now blocks
any single-locale drift at the lint boundary. This is the Wave 26 doctrine
("system grows organically from practice but with structural guarantees")
made operational in two consecutive sub-Wave landings.
