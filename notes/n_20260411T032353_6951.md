# User-side canon misremember: 4-step vs 7-step metabolism pipeline

在 v1.2.0 落地后的第一轮对话里，Yanjun 凭印象说"代谢管道是 4 步吧，之前辩论决定的"，质疑我在 §8.0 写的"7 步管道"。

交叉验证结果：canon 明确锁死 7 步（MYCO.md L27 / vision_recovery_craft_2026-04-10.md L195/337 / _canon.yaml vision_anchors seven-step-pipeline group）。Yanjun 是把"消化道四件套 eat/digest/view/hunger"记混成了代谢管道的步数。事后确认："我记混了，是消化道四件套"。

这是一个有价值的摩擦信号，但属于一个当前 §5.1 没有覆盖的类别：**用户侧 canon 误记 / 用户自承记错**。§5.1(c) on-self-correction 只覆盖 agent 在输出里承认错误，不覆盖这种"人脑记忆 vs canon"的对齐事件。

为什么值得捕获：
1. 创始人本人都会记混 canon 条目 → 说明某个表达不够钉死 → L9 Vision Anchor lint 的必要性被再次印证；
2. "齿轮 4 + 命令 4 + 管道 7" 三个数并存，数字相近且概念相关，是天然的混淆源；
3. 可以作为 Gear 4 candidate 的素材：哪些 canon 条目最容易被记混？→ 指向需要更强 pin / 更好命名 / 或合并的地方。

不立刻写进 §5.1 的原因：v1.2.0 刚签完，立即追加触发点会触发 §8.7 禁止的 bootstrap recursion。正确路径是先 eat 成 raw，等第二个同类样本再考虑 promote。
