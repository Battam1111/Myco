## What fired this wave

Wave 36 §4.4 friction candidate (a) — README still says "15-dimension lint
green" while substrate is at L18 (19 dimensions). Pre-Wave-37 friction scan
discovered the rot was not just in README but throughout the doctrinal
narrative surface: 13 files / ~25 occurrences of stale dimension counts,
stale contract version footnotes, stale verb tables, and a stale §任务 4
"v2.0 ▱▱▱" claim that contradicted Wave 35's MVP scaffold landing.

The reflex arc fired because:
1. **README badge** (`Lint-15%2F15 green`) is the first thing a new user
   sees on GitHub. Stale = first-impression damage.
2. **MYCO.md §任务 4** said `📐 v2.0 ▱▱▱`, but Wave 35 already shipped
   `myco inlet` MVP scaffold under contract v0.27.0. Substrate's primary
   self-description was lying about its own state.
3. **kernel_contract trigger surfaces** (mcp_server.py docstring + tool
   description + cli.py help strings) advertised "15-dimension immune
   system" / "L0-L14" to MCP clients. Substrate was misrepresenting its
   own checking surface to external integrations.
4. **Three READMEs** (English / 中文 / 日本語) were drifting independently —
   the translation mirrors had compounded the rot, not corrected it.

## Decisions D1–D8 (binding for Wave 37 + future similar sweeps)

### D1 — Single source of truth for LINT_DIMENSION_COUNT is `len(lint.py::checks)`
**Why**: Wave 37 audited every surface against `len(checks) == 19` in
`src/myco/lint.py`. Every other surface (README badges, MYCO.md indicator
panel rationale, immune.py docstring, mcp_server.py tool description, cli.py
help strings, CONTRIBUTING.md, wiki/README.md, scripts/myco_init.py shim
description) is **derived** and must be treated as a downstream cache of the
canonical count.
**How to apply**: Whenever a new lint dimension lands, the contract bump
craft MUST include a "label sync" landing item that scans these surfaces and
bumps them all in the same wave. No exceptions. (Wave 38+ candidate: a
LINT_DIMENSION_COUNT-from-ground-truth lint that auto-fails when these
surfaces drift — see §4.4 of Wave 37 craft.)

### D2 — Doctrinal-narrative surfaces (README, MYCO.md §, CONTRIBUTING) are not pinned
**Why**: Hard Contract #2 pins `log.md`, `contract_changelog.md`,
`primordia/*.md`, `notes/*.md`, and `examples/ascc/` as append-only / non-
editable. README, MYCO.md, CONTRIBUTING.md, wiki/README.md, and
docs/{architecture,vision,theory,reusable_system_design,adapters/README}.md
are NOT pinned — they are **living narrative surfaces** that must be edited
to reflect substrate state. Treating them as effectively-pinned was a silent
contributor to the staleness rot.
**How to apply**: When a wave lands a substrate change, ask: which living
narrative surfaces lie about the change? If any, bump them in the same wave
or the immediately following sweep wave. Never let the gap exceed 1 wave.

### D3 — Translation mirrors must always land in the same wave as the English source
**Why**: README_zh.md and README_ja.md had drifted independently from
README.md across multiple waves. Wave 37 had to chase three parallel updates,
which is 3x the surface area for divergence. The root cause is that previous
waves edited only the English README (or failed to edit any of them) and
deferred translation to "later".
**How to apply**: Any edit to README.md MUST be accompanied by mirror edits
to README_zh.md and README_ja.md in the SAME wave / SAME commit. Future
candidate (D7 below): a `lint_translation_mirror` dimension that fails when
the three READMEs diverge by more than N lines.

### D4 — Contract version references in body prose are inherently fragile; keep them only at the SSoT
**Why**: Sweep found `v0.8.0` references in CONTRIBUTING.md, scripts comments,
docs/reusable_system_design.md, docs/adapters/README.md — all from the
v0.8.0 era (5+ contract bumps stale). Each one is a potential lie that
silently degrades trust.
**How to apply**: Body prose should NOT inline contract version numbers
unless the prose is explicitly about a historical state ("as of contract
v0.8.0, X was Y"). Where a current version is needed, point at
`_canon.yaml::system.contract_version` (or its mirror in MYCO.md
metadata) as the SSoT and let the reader resolve. Wave 37 dropped multiple
inline `(L0-L14, contract v0.8.0)` parentheticals to the simpler `(L0-L18)`
form to avoid re-rotting on next bump.

### D5 — kernel_contract trigger surfaces (mcp_server.py + cli.py help) require a craft for ANY edit
**Why**: Even a "label sync" edit to mcp_server.py docstring or cli.py
parser help string touches the kernel_contract trigger surface. The craft
reflex arc (Wave 12) fires. Wave 37 satisfied the reflex by writing the
craft document BEFORE the edits. This is the correct behavior — the craft
can be a "label sync" craft, but it must exist in the same session.
**How to apply**: Future label-sync waves must always start with a craft
document (even a minimal one), even if the only changes are docstring/help
string bumps in trigger-surface files. Do NOT defer the craft to "after the
edits" — the reflex arc's purpose is to capture the reasoning BEFORE the
mutation lands.

### D6 — In-execution scope expansion is permitted when same-root-cause + same-risk-profile
**Why**: While editing MYCO.md per the Wave 37 craft's §1.3 explicit list,
discovered additional stale contract version references at MYCO.md lines 4,
22, 136, 153, 154, 211, 212 that weren't in the craft's original §1
inventory. Made a judgment call to expand scope mid-execution because (a)
same root cause "SSoT label not synced with reality", (b) same risk profile
"narrative-only edit, no behavior change", (c) same file already being
edited. Same logic applied to discovering mcp_server.py line 189 (missed in
first sweep), docs/reusable_system_design.md line 197, and
docs/adapters/README.md line 14.
**How to apply**: Future sweep-style waves may expand scope in-execution
when the new surface is (1) same root cause as a planned surface, (2) same
risk profile (narrative / non-behavioral), and (3) discoverable from a
deterministic grep that any reader could rerun. Otherwise scope expansion
must defer to the next wave with a fresh craft.

### D7 — Wave 38 friction candidates from Wave 37 evidence
**Why**: While sweeping, three latent friction signals surfaced that should
become Wave 38+ candidates per Wave 26 D3 friction-driven ordering:
1. **LINT_DIMENSION_COUNT-from-ground-truth lint** — auto-detect when
   downstream caches drift from `len(lint.py::checks)`. Would have caught
   Wave 37's entire root cause at Wave 30 (when L18 landed).
2. **Translation-mirror lint** — auto-detect when README.md, README_zh.md,
   README_ja.md diverge by more than a threshold. Would have caught the
   per-mirror drift before it reached 3x the surface area.
3. **Contract-version-inline lint** — flag any non-changelog body prose that
   inlines a `contract v0.X.Y` string, suggesting it should point at the
   SSoT instead. Would have caught D4's drift at the source.
**How to apply**: Wave 38 craft should pick the highest-leverage of these
three based on which has the broadest reach into future waves. My current
ranking: (1) > (2) > (3) by reach, but (3) is the cheapest to implement.

### D8 — Wave 37 closes WITHOUT contract bump
**Why**: Staleness sweep is a "label sync" wave. It changes no behavior, no
schema, no hard contract clause, no Hard Contract block, no semantic
guarantee. The kernel contract version stays at v0.28.0. The Wave 26 D3
friction-driven ordering survives intact.
**How to apply**: Future label-sync waves should also close without a
contract bump UNLESS they introduce a new lint dimension or modify the Hard
Contract block. The contract version is for substrate-shape changes, not
narrative-surface refreshes.
