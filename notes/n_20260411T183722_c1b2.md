# Forage digest — hermes-agent (NousResearch/hermes-agent, MIT)

Source: `forage/repos/hermes-agent/README.md` (177 lines) + `AGENTS.md` (469 lines) + file-tree scan of top-level directories (1563 files total). Forage item `f_20260411T180416_3654`. Acquired 2026-04-11 as part of Wave 9 first-live forage batch. Foraging intent: pressure-test Myco's Hard Contract against a mature agent framework's session/identity/tool-use design.

## What hermes-agent actually is

A production-grade open-source AI agent runtime from Nous Research. It's a *runtime*, not a *substrate*: synchronous conversation loop in `run_agent.py::AIAgent`, tool orchestration in `model_tools.py`, session persistence via SQLite + FTS5 in `hermes_state.py`, message gateway for Telegram/Discord/Slack/WhatsApp/Signal/Email, 40+ tools, 6 terminal backends (local/Docker/SSH/Daytona/Singularity/Modal), 3000 pytest tests. Claim: "self-improving" via agent-curated memory + autonomous skill creation + FTS5 session search + Honcho dialectic user modeling. Compatible with agentskills.io open standard. MIT licensed, ~1050 lines in `mcp_tool.py` alone, six release notes files for v0.2 through v0.8 — indicating rapid iteration.

## The Hard Contract pressure test

**Myco's Hard Contract (v0.9.0)** has four clauses: unsure → eat, no tool bypass, friction always captured, kernel changes go through craft + upstream. These are *substrate hygiene* invariants — they govern how durable state evolves across sessions.

**Hermes's equivalent "policies" in `AGENTS.md §Important Policies`** are different in kind: "prompt caching must not break", "don't alter past context mid-conversation", "don't change toolsets mid-conversation", "don't reload memories mid-conversation". These are *runtime performance* invariants — they govern what can change within a single LLM turn chain without breaking the cost model. Different layer.

The two contracts **don't compete, they stack**. A hermes-agent instance running on top of a Myco substrate would be coherent: hermes enforces "don't break cache inside a turn", Myco enforces "don't break metabolism across turns". Neither substitutes the other. This is a useful result from the pressure test: **Myco's Hard Contract is not missing anything that hermes has in its runtime layer, and vice versa**. They're solving adjacent problems.

## The three ideas Myco should weigh (with the 2nd-signal note)

**1. Central command registry as a single-source emitter (gbrain was signal #1, hermes is signal #2).** Hermes's `hermes_cli/commands.py::COMMAND_REGISTRY` is a list of `CommandDef` objects. From that one list, the project auto-derives: the CLI dispatcher, the gateway dispatcher, the gateway help generator, the Telegram BotCommand menu, the Slack subcommand map, autocomplete, and the help-by-category view. Adding an alias is a one-line edit to `CommandDef.aliases`; no other file changes. **gbrain had essentially the same pattern in `src/core/operations.ts` (CLI + MCP generated from 30 shared ops)**. Two independent mature projects converged on this. For Myco this is no longer an abstract observation: *when Myco adds MCP tools (for Cowork surfacing), the verb definitions must come from one contract — probably a section in `_canon.yaml` — not from imperative `click` decorators scattered through `src/myco/cli.py`*. This is now a *Wave-10-or-wave-after-MCP consideration*, not a "someday". **Log priority raised from #nuwa-digest "note-level flag" to "pre-factor-when-MCP-lands"**.

**2. SQLite + FTS5 as the eventual retrieval backend (gbrain was signal #1, hermes is signal #2).** gbrain went Postgres + pgvector at scale, but also supports "just markdown files, no DB" for starting out. Hermes uses SQLite + FTS5 for session persistence and cross-session search. Two different agent projects, both chose SQLite-FTS5 as the default local-scale text retrieval. **For Myco this sharpens the "when retrieval is needed" plan from gbrain digest — when the time comes, it should be SQLite FTS5 + optional duckdb for embeddings, rebuildable from `notes/` + `forage/_index.yaml`**. Reconfirmed, not a new idea.

**3. Profile-scoped state via environment variable (new, 1st signal).** Hermes supports multi-instance via `HERMES_HOME` env var set by `_apply_profile_override()` *before module imports*, and 119+ call sites of `get_hermes_home()` auto-scope to the active profile. Hardcoding `~/.hermes` breaks profiles — this was literally the source of 5 bugs in one PR (#3575 per AGENTS.md). **Myco's mapping**: Myco already supports `--project-dir` as a runtime argument and walks up for `_canon.yaml`. That's *similar* but weaker — it's a per-command flag, not a process-global env var, and you can still accidentally reference hardcoded paths. This isn't a pain point for Myco yet (no known bugs from it), but *if Myco ever wants "hermes-style multi-instance profiles where the same binary manages several `_canon.yaml` roots simultaneously"*, the env-var-set-before-imports pattern is the proven shape. Low priority, file as a possible future direction.

## What's explicitly NOT portable

**The runtime itself.** Hermes's 3000-test pytest suite, 40-tool inventory, 6-backend terminal, message gateway for 6 platforms — this is *a product*, not a library. Myco should not try to become a runtime. Myco should *stay a substrate* and let runtimes like hermes sit on top of it. This is the core strategic read of the forage: hermes makes the "hermes-agent-on-Myco" integration legible by showing exactly where the layers would meet (hermes's `hermes_state.py` session DB would live underneath Myco's `notes/`, hermes's skill system would absorb from Myco's `extracted` notes, hermes's memory-nudges would feed Myco's `eat`).

**The "self-improving" claim architecture.** Hermes's in-process learning loop (curate memory → nudge persist → spawn skills → search history → model the user) is the closest external analog to Myco's Gear 4, but it happens in one process without a durable substrate contract. Myco's version is cross-session and lint-gated. These should not be merged — the in-process mechanism is brittle by comparison, and merging would introduce a runtime-performance tension into a substrate-hygiene layer. Keep them separate; the integration story is "hermes does runtime learning, Myco does substrate compression".

## What Myco has that hermes doesn't

- **Lifecycle metabolism with lint gate.** Hermes's memory is a grab-bag of files in `~/.hermes/` — no raw/digesting/extracted/integrated stages, no compression discipline, no lint pressure. It's durable but not *alive* in Myco's sense.
- **Craft protocol for kernel contract changes.** Hermes ships kernel changes in release notes (`RELEASE_v0.5.0.md` etc), not via a rounds/confidence/decision-class doc. Fine for a runtime, insufficient for a substrate that's meant to compound across projects and agents.
- **Forage substrate with license-aware quarantine.** Hermes ingests skills from `agentskills.io` without any license gating visible in AGENTS.md. At hermes's distribution scale this might be fine (skills are user-curated); for Myco's cross-project compounding it wouldn't be.
- **Upstream absorb (outbox → inbox) primitive.** Hermes has `hermes claw migrate` for one-shot migration from OpenClaw, but no ongoing bidirectional contract-absorption loop from downstream users back to the kernel.

## Convergence summary (the value of doing this forage)

With three digests now in hand (nuwa-skill + gbrain + hermes-agent), a pattern is emerging across independent mature projects:

- **Single-source contract file → many emitters** (gbrain ops.ts, hermes CommandDef). Myco should pre-factor when MCP tools land.
- **SQLite FTS5 is the default local retrieval shape at small-to-medium scale** (gbrain fallback, hermes session DB). Myco's eventual retrieval should follow this.
- **Durable external memory layer is a genuine primitive** (nuwa's research/*.md, gbrain's markdown SoT, hermes's `~/.hermes/`). All three independently chose *markdown files as durable state*. This retroactively validates Myco's core `notes/*.md` decision — it's the convergent choice.
- **Skills-as-markdown is becoming the standard shape** (nuwa's SKILL.md, gbrain's GBRAIN_SKILLPACK.md, hermes's `agent/skill_commands.py`, agentskills.io standard). Myco's integration story with these projects is via the `.claude/skills/` substrate they all share.
