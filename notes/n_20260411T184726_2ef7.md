# Forage digest — mempalace (milla-jovovich/mempalace, MIT)

Source: `forage/repos/mempalace/README.md` (732 lines) + `AGENTS.md` (78 lines) + file-tree scan. Forage item `f_20260411T180425_2146`. Acquired 2026-04-11 as part of Wave 9 first-live forage batch. Foraging intent: look at spatial/mnemonic retrieval primitives and whether their notion of "place" maps onto Myco's forage/notes/upstream three-channel geography.

## What mempalace actually is

A local-first memory layer for AI conversations. Storage is raw verbatim text in ChromaDB. Retrieval is organized by a spatial schema borrowed from classical mnemonics: **wings** (projects or people), **rooms** (topics within a wing), **halls** (memory-type corridors: facts/events/discoveries/preferences/advice), **tunnels** (cross-wing topic links), **closets** (summaries pointing to), **drawers** (the actual verbatim files). Ships an MCP server with 19 tools, a knowledge-graph layer with temporal validity windows (SQLite-backed, Zep-Graphiti alternative), an AAAK "lossy abbreviation dialect" for context packing, `mempalace wake-up` that loads ~170 tokens of critical identity/facts. Claim: **96.6% R@5 on LongMemEval** in raw mode (independently reproduced), which the README itself says is *higher than any published system*, free or paid. MIT, very young (v3.0.0, launched April 2026), community caught real errors in the first 48 hours and authors published a public correction.

## The philosophical tension — taken seriously

Mempalace's core bet is the **opposite** of Myco's: "No AI decides what matters — you keep every word, and the structure gives you a navigable map instead of a flat search index. Store everything, then make it findable." The empirical case is strong: AAAK compression regressed 12.4 points vs raw verbatim on LongMemEval (84.2% vs 96.6%), and the 96.6% headline number is from **raw mode**, not from any of their compression variants.

This is not a cheap-shot critique of Myco. It's a real architectural alternative and deserves a clean response.

**Why Myco's compression bet is still coherent despite mempalace's data:**

1. **Different target metric.** LongMemEval measures *retrieval recall*: given a question, is the answer in the top-K retrieved chunks? Myco doesn't optimize for retrieval recall. Myco optimizes for *agent-behavior change* across sessions: reading the substrate at session-boot should change what the agent does next. There is no published benchmark for that target, so mempalace's numbers can't falsify Myco's bet — they answer a different question.

2. **Myco also keeps raw at the ingress.** `notes/*.md` with `status: raw` are verbatim. `myco excrete` is rare and requires a reason. Compression happens in a staged lifecycle (raw → digesting → extracted → integrated), not as destructive ingestion. A raw note is queryable forever. In that sense **Myco and mempalace agree on "raw survives ingestion"** — they disagree on whether *compressed derivative* artifacts are worth producing alongside the raw tier.

3. **Mechanism difference.** AAAK is *lexical* compression — entity codes, abbreviations, sentence truncation, a regex-based dialect. Myco's compression is *semantic* — synthesis into higher-density forms with explicit rationale and cross-references. The LongMemEval regression is evidence that lexical compression hurts recall; it's silent on whether semantic compression does.

4. **Mempalace's own honest correction validates the craft protocol idea.** The "Milla & Ben note" admits four concrete overclaims that the community surfaced in 48 hours: the AAAK token-count example, the "30x lossless" framing, the "+34% palace boost" framing, the "contradiction detection" wiring status. That's exactly the shape of failure that Myco's craft protocol (rounds, attack, confidence thresholds) is designed to prevent. **Mempalace retroactively validates Myco's insistence on adversarial peer review before shipping claims**.

Conclusion: take mempalace's raw-storage bet seriously as an alternative, note Myco's divergence is *principled* (different metric, same raw-survives policy, semantic ≠ lexical compression), and borrow what's portable.

## The three borrowable ideas

**1. Spatial retrieval schema — wings / rooms / halls / tunnels.** This is the most novel design element in mempalace. It takes classical mnemonics and operationalizes them as retrieval filters. The 34%-recall-improvement-from-wing+room-metadata is "just" metadata filtering on ChromaDB (mempalace's own correction note says this), but the *schema itself* is useful regardless of whether it's magic. **Myco's mapping**: Myco currently has flat `notes/*.md` organized only by tags. A three-level scheme — wing (project or person, single for Myco = "self"), hall (note class: friction, craft-conclusion, forage-digest, retrospective, decision), room (topic cluster) — would give retrieval a navigable structure without changing the underlying file layout. Halls already exist implicitly in Myco via tags (`friction-phase2`, `craft-conclusion`, `forage-digest`). Rooms are latent and would need clustering. **Low-priority but concrete**: file as a "when Myco adds retrieval, also add hall/room metadata in frontmatter" candidate, same bucket as the SQLite FTS5 decision.

**2. Temporal-validity knowledge graph.** Mempalace's `knowledge_graph.py` stores entity-relationship triples with `valid_from` / `ended` windows and supports `as_of` queries. This is a real primitive — "what was true in January" vs "what is true now" is a distinct question from "find me the note about auth". **Myco's mapping**: Myco has nothing like this. Notes carry `created` and `last_touched` timestamps but there's no way to query "what was my position on X three months ago". The `log.md` timeline is append-only prose, not queryable. **Observation**: this is interesting for Myco's longitudinal use case (Gear 4 cross-session learning needs to know "what did I believe last wave vs this wave") but requires a real graph layer. Not actionable at current scale. File as a Gear-4-scale consideration, probably post-v1.0.

**3. Wake-up command loading ~170 tokens of critical facts.** `mempalace wake-up` loads L0 (identity, ~50 tokens) + L1 (team/projects/preferences, ~120 tokens) at session start, before any query fires. **Myco's mapping**: Myco already does this via CLAUDE.md/MYCO.md hot-zone reading at session boot. *Convergent solution*. The sharper idea from mempalace is the **L0/L1/L2/L3 layering**: L0 always-loaded identity, L1 always-loaded facts, L2 on-demand topic recall, L3 on-demand semantic search. Myco's hot zone already plays the L0+L1 role. Myco doesn't yet have formal L2/L3 tiers — `myco view` is a linear scan that treats all notes equally. **Observation, not action**: the layering model could retrofit onto Myco's existing boot sequence as documentation rather than code — "hot zone = L0+L1, `myco view --status extracted` = L2, future retrieval = L3".

## What's explicitly NOT portable

**AAAK itself.** The empirical case against lexical compression on retrieval metrics is clear enough that Myco should not build a Myco-flavored AAAK. If Myco ever wants to save tokens in a hot context, the answer is semantic compression into integrated notes (already the plan), not regex-based abbreviation of raw text.

**ChromaDB as storage.** Myco's markdown-files-are-SoT commitment is incompatible with a vector-DB-as-source-of-truth. Mempalace's own hybrid (files on disk + Chroma index) is fine for a memory layer but imposes an availability dependency Myco doesn't want.

**The "wings = people or projects" model.** Mempalace assumes multi-project, multi-person scope from day one. Myco is explicitly single-project-at-a-time (see friction note `381c` — multi-project mount ≠ multi-project focus). The wing primitive would import a scope confusion Myco just worked to eliminate.
