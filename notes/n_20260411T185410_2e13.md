## What Karpathy's gist actually proposes

A pattern — not an implementation — for building a personal knowledge base where an LLM agent incrementally writes and maintains a persistent wiki of markdown files sitting between the user and the raw sources. The contrast is with retrieval-at-query-time RAG systems (NotebookLM, ChatGPT uploads) where every question re-derives synthesis from scratch; the wiki approach compiles once, keeps current, compounds over time. The LLM owns the wiki files entirely; the human curates sources, asks questions, directs analysis. Obsidian is suggested as the browsing surface, the LLM as the maintainer, the wiki as the codebase-analog.

**Three architectural layers** (Karpathy's framing): raw sources (immutable), the wiki (LLM-written markdown), the schema (a `CLAUDE.md`/`AGENTS.md`-style config file that encodes conventions and workflows).

**Three operations** (Karpathy's framing): Ingest (LLM reads a new source, writes a summary, updates cross-referenced pages, appends to the log), Query (LLM searches wiki, synthesizes answer with citations, optionally files the answer back into the wiki), Lint (periodic health-check for contradictions, stale claims, orphan pages, missing cross-references).

**Two special files**: `index.md` (content-oriented catalog) and `log.md` (append-only chronological record with a parseable entry-header convention for grep-ability).

**Optional tooling mention**: `qmd` is named as a local markdown search engine with hybrid BM25/vector retrieval and an MCP-server interface.

**Intellectual ancestor cited**: Vannevar Bush's Memex (1945) — the part Bush couldn't solve was maintenance; the LLM handles that.

## The `--why` question: does Wave 9's `absorb` extend Karpathy's pattern or drift from it?

This is the core question the digest exists to answer. Verdict: **Wave 9's upstream `absorb` is a genuine extension of Karpathy's pattern, not a drift.** Defense in four moves.

**Move 1 — Myco implements Karpathy's three ops, plus three more.** Mapping the operation surface:

- Karpathy **Ingest** ↔ Myco `eat` (capture a raw source into the substrate) *plus* Myco `digest` (transition raw → digesting → extracted → integrated with compression discipline). Karpathy conflates reading-and-filing into one operation; Myco splits them because the compression lifecycle is the thing that makes the substrate compound rather than bloat. **This is a refinement, not a drift.**
- Karpathy **Query** ↔ Myco `view` (filter/read notes by status, tags, stage). Myco's query surface is thinner than Karpathy's because Myco's scale target is smaller (designed for ~100s not ~10,000s of notes, per the mempalace digest's philosophical exchange). At larger scale Myco would need proper retrieval, and the gbrain+hermes+mempalace convergence points at SQLite FTS5 as the likely default.
- Karpathy **Lint** ↔ Myco `lint` (the 15-dimension L0-L14 gate). Myco's lint is more prescriptive than Karpathy's — it's a committed CI gate, not a periodic suggestion. The dual-path shim (`myco lint` + `scripts/lint_knowledge.py`) exists so that downstream projects can enforce the kernel contract without installing the kernel. Karpathy's lint is advisory and human-in-the-loop; Myco's is a tripwire.

Plus three operations Karpathy's pattern has no equivalent for:

- Myco **forage** (license-gated inbound acquisition from external repos/papers/articles with quarantine primitive). Karpathy's pattern assumes sources are already curated and dropped into the raw collection by the user. Myco's forage family (`add/list/digest/quarantine/discard`) exists because cross-project compounding requires a license-provenance layer that Karpathy doesn't need — his pattern is for a single private wiki, not a substrate that will have commits pushed to a public kernel repo.
- Myco **digest** (as a distinct lifecycle verb, see Move 1 above). Elevated separately here because it's not just Ingest-with-more-steps — it's the metabolic discipline that lets raw survive alongside extracted alongside integrated so that the substrate doesn't lose information while compressing. Karpathy's wiki has no raw tier after ingest; the source is the raw tier.
- Myco **upstream absorb** (cross-instance backflow from downstream project instances to the upstream kernel definition). Karpathy's pattern is single-wiki; there is no notion of "the pattern itself" having a kernel-and-instances shape. Myco has this shape because Myco is both a pattern and a kernel codebase — the ASCC project is a downstream instance running on the Myco substrate, and when ASCC discovers a kernel-contract weakness (friction note 381c was exactly this), the fix needs to flow back to the kernel definition. Karpathy's pattern offers no primitive for that because his wiki doesn't *have* a kernel definition distinct from its instance.

**Move 2 — Myco already converged on Karpathy's two-special-files shape.** `index.md` and `log.md` in Karpathy's gist map directly to Myco's existing `log.md` append-only timeline with the `## [YYYY-MM-DD] type | description` header convention (grep-parseable, same shape as Karpathy's example). Myco's `MYCO.md` dashboard plays a hybrid role — it's partially content-index (indicator panel, hunger signals) and partially the schema file (hard contract, boot protocol). Since Myco's note surface is flat with frontmatter tags rather than a page-and-cross-reference graph, the `index.md` catalog role is currently served by `myco view` queries rather than a static file. **This is a deliberate shape match on the two primitives Karpathy calls out.**

**Move 3 — Myco's four-layer knowledge architecture is the direct descendant of Karpathy's three layers, with raw-source tier split.** Karpathy: raw / wiki / schema. Myco (in the ASCC project context, per `CLAUDE.md` §knowledge system architecture): L1 index (`CLAUDE.md`) / L1.5 wiki (`wiki/*.md`) / L2 docs (`docs/*.md`) / L3 code / L0 archive. The mapping is: Karpathy schema ↔ Myco L1 (`CLAUDE.md` contract), Karpathy wiki ↔ Myco L1.5 + L2 (the extracted/integrated tier), Karpathy raw sources ↔ Myco L0 archive (session history) + forage raw tier. The split between "raw sources" and "raw notes" is Myco-specific and reflects the fact that Myco is built on an agentic session loop, so raw-context-from-conversations is itself a source type distinct from externally-foraged documents. **Acknowledged ancestry**: this layering was first compiled into `docs/current/llm_wiki_debate_2026-04-07.md` in the ASCC project, which is now migrated to the Myco kernel repo under the same path. Karpathy's gist is directly named as the inspiration there.

**Move 4 — MCP-server mention is the 5th signal for MCP-as-agent-surface.** Karpathy's one-line reference to `qmd` having an MCP server lands alongside nuwa, gbrain, hermes, and mempalace as independent projects pointing at MCP as the agent-facing tool surface. The count: nuwa as Claude Code skill (MCP-adjacent), gbrain MCP server, hermes `mcp_tool.py` + full framework, mempalace 19-tool MCP server, Karpathy gist naming MCP explicitly. **Five signals is well past craft-promote threshold.** Confirming the Wave 10/11 escalation from the mempalace digest: the next Myco wave that introduces MCP tools should pre-factor through a single `_canon.yaml` ops contract (2nd-signal from gbrain+hermes central command registry pattern).

## Philosophical divergences — where Myco does *not* follow Karpathy

**Divergence 1 — Obsidian as the browsing surface.** Karpathy's suggested workflow pairs the LLM agent with Obsidian side-by-side. Myco is explicitly filesystem-native and tooling-agnostic: `notes/*.md` are flat markdown files readable in any editor, and Myco currently has no graph-view equivalent because Myco's notes are tag-linked rather than wikilink-linked. This is a conscious trade — wikilinks would require an additional parser and a link-consistency lint dimension (L15 territory), and Myco's scale doesn't yet justify it. **Not a rejection of Obsidian**, just a choice not to depend on a specific editor.

**Divergence 2 — "You never write the wiki yourself."** Karpathy frames the user's role as curator-only. Myco allows the user to write directly into `notes/` (after Wave 9's L11 no-scratch enforcement, anything not in `notes/_templates/` is still subject to the hard contract, but `myco eat` accepts any source including hand-written markdown). This is because Myco's downstream projects include research projects (ASCC) where the human is also the author of primary material, not just the curator of external sources. **Myco is a superset here**: can be run Karpathy-strict (never human-written) or in hybrid mode.

**Divergence 3 — "Compiled once, kept current."** Karpathy's pattern treats the wiki as a living document that gets updated in place. Myco's lifecycle (raw → digesting → extracted → integrated → excreted) is more like a version history with stages — integrated notes are rarely re-edited; instead, new extracted notes cite and refine prior integrated notes, and `deep-digested` (digest_count≥2) becomes a `promote_candidate` signal for craft-protocol review. This is a **principled** divergence: Myco bets that the durable substrate benefits from an audit trail of how compression happened, not just the final compressed form. Karpathy's pattern bets that the final form is what matters and history lives in git. Both can be true; Myco's bet is more conservative.

## What's explicitly NOT portable

**The `qmd` dependency.** Even though MCP signal count is high, pulling `qmd` specifically as a dependency is not the right move — Myco should build its eventual retrieval layer from the convergent SQLite FTS5 pattern, not adopt a specific external search binary. `qmd` is worth knowing about as a reference implementation, not as a dep.

**Obsidian plugins (Dataview, Marp, Web Clipper).** These are editor-specific and Myco is editor-agnostic. If a downstream Myco project wants to use Obsidian, that's fine; the kernel shouldn't assume it.

**The "batch-ingest with less supervision" mode.** Karpathy offers this as a workflow option. Myco's hard contract makes batch-ingest without supervision dangerous — the L11 no-scratch rule, the friction-always-captured rule, and the craft-protocol-for-kernel-changes rule all assume human-in-loop at least at the lifecycle transition points. **Myco requires human-in-loop at `digest --to integrated` and at `upstream absorb`**, by design.
