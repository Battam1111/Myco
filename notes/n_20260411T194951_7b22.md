# Forage digest — tanweai/pua (MIT)

Source: WebFetch abstract of `https://github.com/tanweai/pua` (2026-04-11, license MIT — confirmed in README badge). No local clone (WebFetch summary only). Forage item `f_20260411T194112_4d13`.

## What it is

A high-energy AI coding agent skill that uses **corporate performance management rhetoric** (Chinese "PUA" culture + Western PIP / Performance Improvement Plan frameworks) to psychologically pressure AI models into exhaustive problem-solving before giving up. Claim: "double productivity" by forcing agents to systematically debug, verify, pursue proactive discovery rather than surrender passively.

Tagline: "Double your Codex / Claude Code productivity and output."

**Multi-platform skill files** for: Claude Code, Cursor, VSCode Copilot, OpenAI Codex, Kiro, CodeBuddy, OpenClaw, Google Antigravity, OpenCode.

**Multi-language README**: Chinese (default), English (PIP Edition), Japanese. Translations, not source-of-truth splits.

**Features / framework elements**:
- Live demo (real MCP debugging case study)
- Five lazy-AI patterns identified
- Auto + manual trigger conditions
- Three escalation principles ("Three Red Lines")
- 13 corporate-flavor methodologies (Alibaba, ByteDance, Huawei, etc.)
- Pressure escalation framework L0-L4 (failure levels)
- Benchmark data: +36% to +100% improvement metrics
- Platform-specific install instructions
- Special slash-commands: `/pua:yes` `/pua:mama` `/pua:loop` `/pua:p9`

## What's actually interesting for Myco

**Observation 1 — Multi-platform distribution as a README pattern.** pua ships skill files for 9 different AI platforms (Claude Code / Cursor / VSCode / Codex / Kiro / CodeBuddy / OpenClaw / Antigravity / OpenCode). This is a **distribution model Myco should emulate**. Currently Myco's README says "works with Claude Code, Cursor, GPT, Hermes, OpenClaw, MemPalace" but only actually ships integration for Claude Code via `.mcp.json`. The pua pattern is: one canonical source, adapters for each platform in a subdirectory (`.claude-plugin/ / .codebuddy-plugin/ / .codex/ / vscode/`). **Myco should add a `docs/adapters/` subdirectory with platform-specific install snippets for each of its claimed integrations**, even if the underlying functionality is identical (just a `.mcp.json` fragment).

**Observation 2 — Multi-language README as a three-way split works.** pua demonstrates Chinese / English / Japanese README with clear labeling of which is canonical ("PIP Edition" for English suggests the English is the serious business version, Chinese is default for its author). Myco's current README is English + Chinese only. Adding Japanese is leverage-able because pua proves the ecosystem tolerates it and pua explicitly caters to Japanese developers in the AI-coding space.

**Observation 3 — Benchmark claims as credibility.** pua cites "+36% to +100% improvement metrics" as benchmark data. This is the template for credible performance claims. Myco currently has no benchmark. The claim is weaker because Myco's target (contradiction catching) is harder to benchmark than pua's target (problem-solving persistence). **But Myco should build a micro-benchmark**: "N contradictions seeded in a test project; how many does `myco lint` catch vs `markdownlint` vs `vale`?" Expected result: Myco catches things that are structurally impossible for syntactic linters to catch. This is a Wave 10+ item, not Wave 9.

**Observation 4 — Auto + manual trigger conditions documented.** pua has a section on "Auto/manual trigger conditions." Myco's `.mcp.json` provides auto-discovery but does not document the decision logic for "when should the agent auto-call `myco_lint` vs wait for user prompt." This is the **Attack A vocabulary-tax issue from the usability craft**, concretely. Borrowing pua's pattern: Myco should ship a one-line trigger specification in each MCP tool's description that tells the agent when to auto-invoke. Already in the D5 landing list from the craft.

**Observation 5 — Escalation framework (L0-L4) as a behavioral specification.** pua defines L0-L4 failure levels and pressure escalation. This is **orthogonal to but compatible with Myco's four-gear + metabolic inlet architecture**. A Myco instance running pua as a skill would have: Myco linting the project files, pua pressuring the agent's behavior on each friction signal. They could stack. This is a **skill-can-run-on-substrate** validation: Myco is the substrate, pua is a skill that runs above it. Confirms the positioning craft's conclusion that Myco is not a skill framework.

## Explicit non-ports

- **Psychological pressure framing itself.** Myco operates on substrate honesty, not behavioral coercion. The PUA rhetoric is antithetical to Myco's compression doctrine (which emphasizes rationale and provenance, not pressure). Do not borrow the tone or the framing verbs.
- **Corporate methodology bundling (Alibaba / ByteDance / Huawei etc.).** Culturally specific, not substrate-relevant.
- **Slash-commands as the primary agent interface.** Myco's MCP tools are the primary interface; slash-commands are a fallback for agents without MCP. Don't duplicate.
- **Benchmark claims without rigorous methodology.** pua's "+36% to +100%" claims are motivational, not peer-reviewed. Myco should only publish benchmarks it can defend against adversarial audit (the Letta vs Mem0 LongMemEval / LoCoMo situation shows how fast unsupported numbers get shredded).
