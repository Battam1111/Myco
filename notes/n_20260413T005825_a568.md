Viral README Playbook — synthesized from 19 sources (2025-2026).

KEY FINDINGS:
1. 3-SECOND RULE: Logo + one-liner + badges + demo GIF must be visible without scrolling. 75% bounce if first screen fails.
2. STATS: Comprehensive READMEs get 4x more stars. Screenshots = +42% stars. Star badges = +15% conversion.
3. PRESCRIBED ORDER: Hero block → Highlights (pain→solution) → Quick Start (3 steps max) → Features → Architecture diagram → Installation → Usage examples → Used By → Contributing → License
4. EMOTIONAL HOOK: Problem-Agitation-Solution pattern. Lead with user's pain, not your story. "Tired of X? ProjectName does Y in Z seconds."
5. DEMO GIF: Single most impactful element. Under 20s, 15fps, 600x400. Place before any text sections.
6. QUICK START: Must work on first try. 3 copy-paste steps. If needs more, it's an onboarding problem.
7. COMPARISON TABLE: Features as rows, projects as columns, checkmarks. Factual, not subjective.
8. README ≠ DOCS: README is a sales page. Cover what and why. Link to docs for how (deep).
9. COLLAPSIBLE SECTIONS: Use details/summary for advanced content.
10. SOCIAL PROOF: Star history chart, "Used By" logos, download counts, contributor badges.
11. RICHARD KIM METHOD: First 30 stars via personal outreach triggers GitHub trending algorithm.
12. README HOURS = CODE HOURS: Spend equal time on README as on code (1:1 ratio).

ANTI-PATTERNS:
- Don't start with "This is a project that..."
- Don't put installation before value proposition
- Don't include empty sections
- Don't use "Last Updated: DATE" (signals staleness)
- Don't write paragraphs where bullets work
- Don't make README >2000 lines (move to /docs)
