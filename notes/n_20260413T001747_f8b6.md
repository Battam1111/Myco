# Distribution model: editable vs normal install for evolving system

User question about distribution model: should Myco always use editable install since it needs to continuously evolve?

Analysis of the two installation modes:

1. **pip install myco** (normal install) — copies code to site-packages. User gets a frozen snapshot. Updates require `pip install --upgrade myco`. This is the standard PyPI distribution model. Pros: simple, reproducible, no git needed. Cons: can't self-evolve without re-publishing to PyPI.

2. **pip install -e .** (editable install from git clone) — symlinks to source directory. Code changes take effect immediately. This is what we use for development. Pros: self-evolution works (evolve.py mutations, skill mutations, template updates all instant). Cons: requires git clone, more complex setup.

The answer depends on WHAT evolves:
- **Substrate content** (notes, wiki, docs, _canon.yaml, skills) — these are per-project, NOT in the pip package. They live in the project directory. Normal pip install is fine for this.
- **Kernel code** (src/myco/*.py) — this is the pip package. Self-evolution of kernel code requires either editable install OR a new PyPI release.
- **Skills** (skills/*.md) — per-project, not in pip package. Normal install is fine.
- **Templates** (src/myco/templates/) — in the pip package. Template evolution requires editable or new release.

Conclusion: for most users, normal `pip install myco` + `myco init` is sufficient because evolution happens at the PROJECT level (notes, skills, _canon.yaml), not at the PACKAGE level. Kernel code evolution is rare and should be published as new PyPI versions. Only Myco developers (us) need editable install.
