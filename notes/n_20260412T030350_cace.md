## The 7 design questions (from Wave 26 §3.3)

Restated here for the decisions note to reference:

1. **Unit**: what does one compression operate on? (tag / time / semantic / manual / status)
2. **Trigger**: how is compression initiated? (manual / scheduled / friction-driven)
3. **Output shape**: what does compression produce? (additive / consumptive / wiki / in-place)
4. **Audit trail**: how does a human verify the compression is not hallucination?
5. **Reversibility**: can a compression be undone? what's the contract?
6. **Step 2-5 interaction**: does compress fold anchor #3 middle steps, or call them?
7. **Non-functional requirements**: idempotence / atomicity / concurrent safety / cost

## Working proposals (Round 1 summary)

- **Q1 Unit**: tag-scoped + manual bundle. Reject semantic cluster (parametric), time-window (too coarse), status-scoped (foldable)
- **Q2 Trigger**: manual + friction-driven (`compression_ripe` hunger signal). Reject cron.
- **Q3 Output**: consumptive with audit (1 new extracted note + N excreted inputs with bidirectional links)
- **Q4 Audit**: mandatory frontmatter fields (`compressed_from` list, `compression_method`, `compression_rationale` prose, `compression_confidence` self-reported); L18 lint enforces bidirectional integrity; cascade rejected at schema time
- **Q5 Reversibility**: data-preserved (inputs on disk as excreted, not deleted) but verb deferred to Wave 29+. Optional `pre_compression_status` frontmatter field
- **Q6 Pipeline**: compress folds steps 2+3+5+7; step 4 (integrate) stays separate; `--dry-run` preserves step 2 observability
- **Q7 NFR**: idempotent (no-op on empty cohort), atomic (temp+rename multi-file), concurrent-safe (snapshot before writes), O(N) cost bound; agent-adaptive deferred to open problems

## 4 attacks that ran (Round 2 summary)

- **Attack F — Hallucination risk**: bounds the risk (reversible, auditable, non-cascading, single-point-audit acknowledged as structural weakness). Confidence target stays 0.85 (not 0.90) because residual risk is real. Registered as L1.
- **Attack G — Fold-into-compress too aggressive**: lands partially — `--dry-run` becomes mandatory to preserve step-2 observability. Fold decision stays.
- **Attack H — Reversibility is theater**: lands — language softened from "reversible by design" to "preserved for reversibility". `uncompress` verb stays Wave 29+.
- **Attack I — Wave 28 scope creep via "declared dependencies"**: lands as "Wave 28's own scope craft must resolve". Recommendation: bundle `myco compress` + minimum atomic-write helper as ONE subsystem, not two.

## Key design commitments for Wave 28 implementation brief

- New verb `myco compress` with `--tag` XOR positional ids, `--rationale` required, `--dry-run` supported
- New frontmatter fields on extracted output: `compressed_from`, `compression_method`, `compression_rationale`, `compression_confidence`
- New frontmatter fields on excreted inputs: `compressed_into`, optional `pre_compression_status`
- New `_canon.yaml::system.notes_schema` additions: `optional_fields` extended, `valid_sources` + `compress`, new `compression` sub-section with `ripe_threshold` + `ripe_age_days`
- New hunger signal: `compression_ripe` (advisory LOW, not HIGH reflex)
- New lint dimension L18: `lint_compression_integrity` (HIGH) — bidirectional link integrity + cascade prevention
- New Python module: `src/myco/io_utils.py` with `atomic_write_text` + `atomic_write_yaml` helpers (~30 lines; the minimum surface the compress operation needs)
- Contract bump v0.25.0 → v0.26.0
- Wave 28 scope = one subsystem: "compress operation + its supporting atomicity"
- NOT in Wave 28: general atomic-write refactor of `notes.py`/`upstream.py`/`forage.py`; `uncompress` verb; agent-adaptive policy; Metabolic Inlet integration
