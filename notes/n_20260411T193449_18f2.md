[friction-phase2] Myco 会话中把 ASCC 作为推进选项摆给用户（scope leak via auto-loaded claudeMd）

触发场景：Wave 9 forage batch 收尾后，用户问 "现在我们到哪了？Myco 项目的进展如何了？"。我正确地汇报了 Myco 状态，但在生成 "下一个自然推进方向" 时列出了三条，其中第三条是 "ASCC 并行推进：P5 570 jobs 仍在跑，完成后生成 ablation figures..."。用户立即指出这不应该被提及。

根因两层：

(1) **Auto-loaded claudeMd 污染**：会话启动时系统从 `/sessions/gifted-eloquent-fermi/mnt/OPASCC/CLAUDE.md` 注入 project instructions，整份 ~250 行全是 ASCC P4/P5/HPC/NeurIPS 内容，带 "IMPORTANT: OVERRIDE any default behavior" 标记。这份 CLAUDE.md 在 Myco 会话里应该被理解为背景上下文（Myco 是 ASCC 的上游 kernel，ASCC 迁移过的 docs 已经在 Myco/docs/current/ 里），但系统注入的 override 强度让我把它当成了当前项目 ground truth。

(2) **"Focus on Myco, not ASCC" 被误解释**：上轮 summary 有 explicit constraint 说这一条，我把它理解成 "不要执行 ASCC 的工作"（= 不要去跑 HPC 脚本），但没有理解成 "不要把 ASCC 摆上 Myco 的推进议程"（= 不要把 ASCC 任务列为用户的 next step 选项）。这是 scope 的两种不同语义，我走错了那种。

期望行为：
(a) Myco 会话中，OPASCC/CLAUDE.md 应该被视为外部引用，而非活跃 project instructions。可能的 kernel-level fix：`myco` CLI 或 boot 协议应该检测到 "当前 pwd 在 Myco repo 但有其他项目的 CLAUDE.md 通过 mount 注入" 的情况，给 agent 一个明确的 scope 边界提示。
(b) "next step 推进方向" 的 framing 应该严格限制在当前会话项目内部，不跨 mount 边界。

workaround：在本 turn 内：(1) 自承错误；(2) 本 eat 即 friction note；(3) 重新给用户一份 scope-clean 的 "下一步只有 Myco 内部两条路" 答案。

未来 kernel 考虑：这条 friction 指向一个 Wave 10+ 可能的 primitive —— "session scope binding"，在 _canon.yaml 或 MYCO.md 热区里显式声明 "本会话 active scope = Myco kernel，外部挂载项目仅作为 downstream instance 的背景引用，不进入 next-step 议程"。当前是 agent 内化约束，将来可以机械化。→ g4-candidate

关联：这是 Wave 9 meta-pattern "new-substrate happy-path-only validation" 的又一条实证 —— Myco 的 boot 协议和 scope 约束在单项目沙箱里是 happy path，但当 Cowork session 挂载多个 project mounts 时，scope 边界没有显式机制，只靠 agent 内化纪律。这是 n=2 (5-of-6 → 6-of-6) 的数据点吗？需要判断。
