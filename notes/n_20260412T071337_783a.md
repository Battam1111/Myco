# Wave 35 decisions — inlet MVP D1-D9 canonical record

Wave 35 — myco inlet MVP — landing decisions (canonical record).

This note carries the canonical decisions D1-D9 from docs/primordia/inlet_mvp_craft_2026-04-12.md and is the integrated landing record for Wave 35.

D1 — Canon section placement: system.notes_schema.inlet (NOT system.inlet at top level). Rationale: notes-domain default tag belongs under notes_schema. Alternative locations would either spread notes-domain config across two top-level keys or bury "what defaults apply" inside "what fields exist".

D2 — Provenance fields are in OPTIONAL_FIELDS, not REQUIRED_FIELDS. Soft L10 enforcement at severity LOW. Rationale: hard requirement would break grandfathering; the Wave 20 silent-fail doctrine requires fail-loud for sensors but the inverse holds here — we want a soft warning that surfaces gaps without blocking.

D3 — Exit codes locked: 0 success / 2 usage (missing/conflicting args, URL form rejected) / 3 input validation (file not found, not regular, not UTF-8) / 5 io (cannot resolve project root, cannot write notes/). No exit 4 because inlet has no empty cohort analog (always produces exactly one note).

D4 — URL detection is prefix-based on positional source via startswith(http://, https://). Edge case: file paths starting with https- are operator-disambiguated via ./ prefix per shell convention. Filed as L7 limitation.

D5 — _resolve_inlet_input is the single source of truth for input resolution. argparse does NO pre-rejection. Rationale: keeps resolution logic testable in isolation; URL-form error message lives next to its rejection logic.

D6 — Mutual exclusion handled at runtime, not via add_mutually_exclusive_group. Rationale: argparse error messages are not operator-friendly; runtime resolution can produce curated errors that point operators at the agent-fetch pipe pattern.

D7 — L10 soft inlet check severity is LOW. Rationale: MEDIUM would aggregate into operator concern count; LOW signals "this is a hint not a violation"; L10 already uses MEDIUM for actual schema violations.

D8 — 5 tests, one per load-bearing path (Wave 34 §3.1 brief). Test count is calibrated to per-decision boundaries, not external benchmark. Each test guards a Wave 34 design decision.

D9 — Body MAY include a header. sha256 is computed over input bytes (the content parameter), NOT over the rendered note body. Future tamper verifiers must re-derive sha256 from the original source, not from the note body.
