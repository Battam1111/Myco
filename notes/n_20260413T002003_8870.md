# Design reversal: editable install as primary distribution

User challenges the kernel/instance separation: "editable install as default could be a selling point" and "normal install reduces what can evolve."

Rethinking: User is right to push back. The Bitter Lesson says don't limit what can evolve. If kernel code is frozen in site-packages, then:
- evolve.py can mutate skills but can't mutate itself
- lint.py can't grow new dimensions without a PyPI release
- hunger signals can't be added without a PyPI release
- The "self-evolving" claim is half true — only content evolves, not the engine

The most honest, most powerful model: ALWAYS clone from git, ALWAYS editable install. This means:
- The ENTIRE system can evolve: kernel code, templates, content, everything
- git provides version control + rollback (safer than pip for evolving code)
- Users get the latest mutations immediately via git pull
- evolve.py can mutate not just skills but also lint rules, hunger signals, templates

The tradeoff: harder setup (git clone + pip install -e) vs simpler (pip install myco). But Myco's target user is an AI Agent, not a human. The Agent can run `git clone` + `pip install -e .` — it's one more command.

Design decision: make git clone + editable install the PRIMARY distribution path. Keep PyPI as a convenience fallback but document it as "frozen snapshot, limited evolution."
