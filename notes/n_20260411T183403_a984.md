## What gbrain actually is

A personal knowledge brain for AI agents: a markdown repo on disk becomes the source of truth, Postgres + pgvector indexes it for hybrid search (vector + keyword + Reciprocal Rank Fusion + multi-query expansion), an MCP server exposes it to any agent, and a "skillpack" (`docs/GBRAIN_SKILLPACK.md`) tells the agent *when* to read/write. Production claim: one user has 10,000+ markdown files, 3,000+ people pages, 13 years of calendar data, 280+ meeting transcripts indexed. Includes recipe-based integration installers (ngrok, Twilio voice, Gmail, Calendar, X). MIT, active, contract-first design in `src/core/operations.ts` (CLI + MCP server both generated from one ~30-operation contract).

## The four ideas Myco should weigh

**1. Contract-first operations file as the single source of truth for CLI + MCP.** gbrain's `src/core/operations.ts` defines ~30 shared operations; both the `gbrain` CLI and the MCP stdio server are *generated* from that file. There is no drift between "what the CLI can do" and "what MCP exposes" because there's one contract and two emitters. **Myco's mapping**: `_canon.yaml` already exists and carries canonical identifiers and constants, but CLI verbs (`eat/digest/hunger/forage/upstream/lint/view`) are defined imperatively in `src/myco/cli.py`. An MCP surface that mirrors the CLI would drift unless the verb set comes from a single contract file. *Right now Myco has only the CLI, no MCP server*, so this isn't an active pain — but when Myco eventually wants to expose `myco_log` / `myco_eat` / `myco_digest` as MCP tools (and it will, for Cowork and other agents), pre-factoring to contract-first avoids a later rewrite. **Worth a note-level flag, not an action**. Log as: "when MCP tools surface is added, pre-factor through `_canon.yaml` ops section".

**2. "Repo is source of truth, retrieval is a separate layer."** gbrain explicitly keeps this split: markdown files are the system of record, Postgres + pgvector is *purely a retrieval accelerator* that can be rebuilt from the repo at any time. Human always wins — if you edit a markdown file directly, `gbrain sync` picks it up and the index regenerates. **Myco's mapping**: Myco already has the first half of this property (`notes/*.md` are the SoT) but *no retrieval layer at all* — `myco view` does a linear scan of frontmatter, `grep` is the current "search". At Myco's current scale (35 notes) this is totally fine. The principle to extract is *negative*: **when a retrieval layer is eventually added, it must be rebuildable from `notes/` with zero data loss, i.e. the index never becomes part of the source of truth**. This protects the "human always wins" property. Don't build it now; commit to the rebuildability constraint now.

**3. Recipe-based integration installers.** gbrain ships integration recipes as markdown files with YAML frontmatter: one recipe per integration (ngrok tunnel, Gmail, Twilio voice, X, Calendar). The agent reads a recipe, asks the user for the credentials it specifies, validates them, runs a smoke test, and writes the result. The recipe is simultaneously human-documentation and agent-installer — "markdown is code". **Myco's mapping**: Myco's Wave 9 `upstream absorb` flow is conceptually similar — it takes a bundle from outside and flows it into the substrate — but it's not "recipe-driven", it's format-driven (bundle schema). An interesting future shape would be **forage-recipe**: "to forage from source type X (arXiv / HN thread / github release notes), here's the per-type instructions". This would let `myco forage add --recipe arxiv https://arxiv.org/abs/...` derive the license, populate metadata, and stage the download with one command. Not urgent. File as a Gear 4-scale pattern, not a Wave 10 item.

**4. The brain-agent READ → respond → WRITE loop as mandatory-on-every-turn.** gbrain's compounding thesis: every agent turn must start with a retrieval query against the brain and end with a write-back of any entity-level updates. The "dream cycle" is a scheduled autonomous enrichment pass that runs while the user sleeps. **Myco's mapping**: Myco's session boot sequence already requires `myco hunger` (a form of retrieval against substrate health), and Myco has Gear 3 retrospective (a form of dream-cycle consolidation). The missing piece is the *write-back guarantee on every meaningful action* — gbrain's claim is "every turn writes something back, even if it's just a last_touched bump". Myco partially has this via `digest_count` bumping on `myco digest`, but not on `myco eat` or on Cowork agent turns in general. Low-priority but interesting: should every agent turn in Cowork automatically eat a one-line "session continuity" note? Probably not (would pollute raw pile) — but *should every Gear 3 retrospective write back a delta?* Yes, and it already does via log.md milestones. So Myco is mostly covered here, just not explicitly framed as a mandatory loop.

## What's explicitly NOT portable

**Postgres + pgvector as a retrieval layer.** At Myco's scale (35 notes today, projected <10k over project lifetime), a dedicated database is infrastructure overkill and adds a failure mode the current file-based system doesn't have. The "markdown is SoT, retrieval can be rebuilt" principle is portable; the specific Postgres choice is not. If/when retrieval is added to Myco, it should probably be SQLite FTS + optional local embedding file (duckdb), not a hosted DB.

**The 10k-file production scale.** gbrain optimizes for a user who has been compiling a personal brain for years. Myco's scale is different: it optimizes for a small, high-quality digested core. The "compression discipline has teeth" principle in Myco (see `forage/README.md`) is the *opposite* design pressure from gbrain's "feed everything in, let search sort it out". This is a real philosophical divergence and should not be papered over.

## What Myco has that gbrain doesn't (symmetric audit)

- **Four-stage metabolism with lifecycle transitions** (raw/digesting/extracted/integrated/excreted). gbrain has no lifecycle — a page is either there or not. Myco's lifecycle is the compression discipline's enforcement mechanism.
- **15-dimension lint as a committed gate.** gbrain has unit/E2E tests, not knowledge-substrate lint.
- **Upstream absorb as a kernel primitive.** gbrain has integration *inbound* via recipes but no backflow from a downstream instance to an upstream kernel — its "upstream" is a one-directional dependency, not a contract-absorption loop.
- **Forage substrate with license-aware quarantine.** gbrain's ingestion has no license awareness — a markdown file is a markdown file. Myco's forage layer differentiates MIT repos from proprietary articles and gates on that.
