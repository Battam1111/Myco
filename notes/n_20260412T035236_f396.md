## Core tension identified

Myco's doctrine claims **Autonomous Cognitive Substrate** (fungal organism metaphor, 6 axioms, theoretical lineage from Karpathy/Polanyi/Argyris/PDCA/Voyager). Half the implementation naming supports this (eat/digest/hunger/forage/primordia/excrete), half actively undermines it (lint/wiki/canon/notes.py/correct/L7 Wiki Format/L10 Notes Schema). Readers walking through the repo get mixed signals about what Myco IS.

Wave 10 vision_recovery §0 original diagnosis: *"The public README positions Myco as a watcher (lint, reflect, reference integrity). The original vision positioned it as a growing organism that eats the world and digests it into shared knowledge. These are not the same thing."* Wave 28 extends this diagnosis from "public README prose" to "implementation-layer naming".

## 4 attacks ran (Round 2 summary)

- **Attack J** — "Renaming is cosmetic breaking change, cost > benefit". Defense: user grant explicit, migration automatable via Wave 29 `myco migrate --biomimetic` tool, rename list already focused (not exhaustive), semantic lift is measurable per-item, the "nice-to-have imposed by over-reading" framing inverts the trust relationship. Lands partially on cost concern — Wave 29 must bundle migration tool.
- **Attack K** — "myco lint should NOT keep its name; CLI-vs-module split is worst of both worlds". Defense analyzed three states (A=all lint/lint, B=lint CLI + immune module, C=all immune/immune). Concluded Myco's primary user is the agent, not the human; agents learn from --help not muscle memory; State C accepted. **This attack drove the biggest revision** — `myco lint` → `myco immune` added to rename list beyond Round 1 proposal.
- **Attack L** — "Wave 29 bundling renames + migration tool + ASCC dogfood violates Wave 25 capacity". Defense: three phases of one atomic transformation (coupled), not three independent subsystems. Lands as "document this reasoning in Wave 29's own Round 2" — which Wave 28 pre-does here.
- **Attack M** — "`_canon.yaml` → `_genome.yaml` confuses new contributors". Defense: no community convention for "_canon.yaml" exists, README pointer + top-of-file comment suffices, the identity lift is strongest on this specific rename. Lands partially, mitigation = top-of-file documentation comment.
