# Sandbox EPERM forgot Windows-MCP/desktop-commander bypass

沙箱 EPERM 时忘了 Windows-MCP / desktop-commander 旁路：Wave 9 收尾阶段，Cowork Linux 沙箱在 OPASCC 挂载点遇到 unlink EPERM（可 create/overwrite，不可 rm），还留下一个卡死的 .git/index.lock。我的默认反应是把后续工作（git push + ASCC chore commit）全部推回给用户手动跑，完全忽略了 Cowork 本身挂了 Windows-MCP__PowerShell 和 desktop-commander__start_process 这两条可以直接在用户本机跑指令的通路。用户点醒后用 desktop-commander 启了一个 powershell.exe PID，interact_with_process 连续跑了 git push origin main（Myco: da5f2d9..95faea0）和 ASCC chore commit（cf31bea），全部一次成功。触发场景：沙箱文件操作命中权限边界，agent 错把『沙箱能力』等同于『Cowork 能力』。期望行为：遇到任何沙箱侧 EPERM / ENOENT / mount 半只读时，先检查 deferred tool list 里有没有 Windows-MCP / desktop-commander / Claude_in_Chrome 这类 host-side bypass，再决定是否降级为『让用户手动跑』。workaround：用户直接授权后我用 desktop-commander start_process + interact_with_process 成功跑通。元教训：能力边界要分层检查（sandbox → host MCP → user），不能直接从最窄一层跳到最宽一层。
