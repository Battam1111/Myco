# Karpathy LLM Wiki — Architectural Convergence with Myco

## Overview (Published April 3, 2026)

Andrej Karpathy published a gist describing an "LLM Knowledge Base" / "LLM Wiki" architecture that bypasses RAG with an evolving markdown library maintained by AI. This architecture is strikingly convergent with Myco's design — independent invention of nearly identical principles.

## Three-Layer Architecture
1. **Raw Sources** — immutable documents (articles, papers, data files) as single source of truth
2. **The Wiki** — LLM-generated markdown files: structured pages with summaries, entities, concepts, cross-references
3. **Schema** — configuration document (e.g., CLAUDE.md) defining wiki structure, conventions, workflows

## Operational Workflows
- **Ingest**: Drop new source -> LLM reads, extracts takeaways, creates summary page, updates index and entity/concept pages, appends to changelog. Single source typically touches 10-15 wiki pages.
- **Query**: Questions directed at wiki -> LLM searches, synthesizes answers with citations -> quality findings filed back as wiki pages
- **Maintenance/Linting**: Regular health checks for contradictions, stale claims, orphaned pages, missing cross-references, knowledge gaps

## Critical Infrastructure Files
- **index.md** — content-oriented catalog listing every page with summaries (Myco parallel: MYCO.md)
- **log.md** — append-only chronological record (Myco parallel: log.md)

## Key Design Decisions
- Git as foundation (version history, branching, collaboration at zero cost)
- Plain markdown for maximum LLM familiarity and human readability
- At ~100 articles and ~400K words, index file navigation sufficient without embedding infrastructure
- Explicitly avoids RAG's point-of-query rediscovery pattern

## Convergence Points with Myco (Critical)
| Karpathy LLM Wiki | Myco |
|---|---|
| Raw sources -> Wiki compilation | notes/ (raw) -> digest -> wiki/ |
| index.md as navigation hub | MYCO.md as entry point |
| log.md append-only history | log.md append-only timeline |
| Linting for contradictions/orphans | 23-dimensional lint (L0-L22) |
| Markdown as source of truth | Markdown + YAML frontmatter |
| Git-based version control | Git-based version control |
| LLM as librarian/maintainer | Agent-First organism model |

## Where Myco Goes Further
1. **Lifecycle metabolism**: Karpathy has ingest/query/lint. Myco has raw->digesting->extracted->integrated->excreted (full digestive lifecycle)
2. **Self-evolving rules**: Myco's craft protocol evolves the system's own rules. Karpathy's schema is static.
3. **Immune system**: Myco's 23-dimension lint is far more comprehensive than Karpathy's contradiction/orphan checks
4. **Compression**: Myco has explicit compression with audit trails. Karpathy has no compression mechanism.
5. **Hunger/metabolism**: Myco senses what it needs (hunger signals). Karpathy's system is passive.
6. **MCP integration**: Myco exposes 19 MCP tools. Karpathy's system is IDE-bound.

## Where Karpathy Has Advantages
1. **Simplicity**: Much simpler architecture, lower barrier to entry
2. **Mindshare**: Karpathy's reach means massive adoption potential
3. **Scaling guidance**: Multi-agent mutual exclusion, naming conventions documented

## Strategic Implication
Karpathy's publication validates Myco's core thesis — markdown-first, LLM-maintained, git-backed knowledge bases are the right architecture. But it also means the idea is now public and will attract competitors. Myco's differentiation must lean hard on metabolism, self-evolution, and immune system — the biological metaphor is not decoration, it is the architecture.
