# Agent 审计结果必须交叉验证

## 教训

批量派 Agent 做审计时，主 Agent 盲目信任子 Agent 的 FLAG/BLOCKER 判断导致错误修复（删除了实际支持的 Cowork 平台声称）。

## 根因分析

1. **Prompt 框架错误**：指令"检查 detect_tools() 是否包含 Cowork"，但 Cowork 不需要 detect_tools()——它是 Claude Code 的 skills 功能，通过 .claude/skills/ 目录工作。正确的检查应该是"init 后该平台是否能实际工作"。

2. **子 Agent 结论未交叉验证**：Agent 说 "BLOCKER: Cowork 是虚假声称"，主 Agent 看到 BLOCKER 就直接派另一个 Agent 去删。没有用自己的项目知识验证。

3. **批量修复加速了错误扩散**：多个 Agent 并行修复时，一个错误判断可以通过"修复"的名义快速传播到多个文件。

## 不可变规则

- Agent 审计结果 = **建议**，不是指令
- 每条 FLAG/BLOCKER 执行前必须交叉验证：用 grep/read 确认事实
- 特别是"删除"类建议——删除是不可逆的，验证成本远低于恢复成本
- Prompt 设计要给子 Agent 足够的领域上下文，避免表层分析导致错误判断
- 子 Agent 的"不存在"结论尤其危险——它可能只是没找对地方
