# L14 forage orphan-check blind to directory subtrees

L14 forage orphan check broke on directory local_path: first-ever live multi-repo forage (Wave 9, 6 items: gbrain + nuwa-skill + hermes-agent + mempalace + 2 articles) surfaced that lint_forage_hygiene's orphan check treated every individual file inside a cloned repo as an orphan, because referenced_paths is built from 'local_path' values directly without any subtree awareness. A manifest entry like local_path=forage/repos/gbrain covered only the directory inode, not the 181 files underneath. Resulting lint noise: hundreds of L14 LOW issues for what should be 0. Fix: added referenced_dirs subset; file is considered covered if any ancestor dir is in referenced_paths. Lines: src/myco/lint.py lint_forage_hygiene ~L1044-1070. Lint is back to 15/15 PASS. 触发场景: forage add with local_path pointing at a directory (normal case for repo source_type). 期望行为: manifest directory entries cover their entire subtree transparently. workaround: fix-in-place during the foraging session itself (dogfood). 元教训: L14 had never seen a real repo forage until now — the forage_substrate_craft spec was correct in principle but the orphan implementation only covered the 'one file per manifest entry' case (articles/papers). Craft validation ran only against single-file fixtures. Next time a new substrate is added, write a smoke test that uses every valid source_type before merging.
