# Wave 9 retrospective — five dogfood-surfaced gaps, one meta-pattern

**Cluster**: 5 raw notes from 2026-04-11 (f153 craft-conclusion + bc0c sandbox-bypass + 3875 L14-subtree + 381c focus-drift + 8ea3 payload-hygiene). Integrated here as one retrospective so downstream readers see the pattern, not the grains.

## What actually happened in Wave 9

Wave 9 shipped `upstream absorb` (contract v0.8.0 → v0.9.0): `myco upstream scan/absorb/ingest`, pointer-note design with `source: upstream_absorbed`, bundle archived under `.myco_upstream_inbox/absorbed/`, L1 context-aware skip (fences + NEG keywords + `l1_exclude_paths`), L12 absorbed-subdir whitelist, new indicator `upstream_inbox_pressure` (ceiling=5). Craft ACTIVE @ 0.91 (see f153). Dogfood round 1 absorbed `ce72` and `3356` from ASCC outbox, both ingested as pointer notes. Round 2 was a live multi-source forage run: 4 repos (gbrain, nuwa-skill, hermes-agent, mempalace) + 2 articles (claude-managed-agents, karpathy LLM wiki). Everything landed in `forage/` and triggered dual-path lint + commit on the spot.

## The five gaps, as they surfaced

**G1 — L14 orphan scanner was flat-file only (3875).** `lint_forage_hygiene` compared each file under `forage/{papers,repos,articles}` against the exact `local_path` set from `_index.yaml`, which only worked for single-file manifest entries. The Wave 9 dogfood cloned four whole repos whose manifest entries point at a **directory** (`repos/<name>`), so every file inside each repo got flagged as an orphan. Fix (same session): added `referenced_dirs = {p for p in referenced_paths if p.is_dir()}` and an ancestor-chain membership check so a file is "covered" if itself or any ancestor is in the manifest. Shipped in `src/myco/lint.py::lint_forage_hygiene`, commit `845ee39`.

**G2 — forage `.gitignore` was asymmetric (8ea3).** `papers/**` and `repos/**` were fully ignored, but `articles/**` only ignored `.html` snapshots, so the default for `.md` articles was "tracked". Until Wave 9 nobody had actually foraged an article, so the asymmetry never hit. In the live run, a proprietary blog extraction and an unknown-license gist landed in `forage/articles/` and would have committed by default. Fix: rewrote `forage/.gitignore` to `articles/**` with `!articles/.gitkeep`, plus README rationale (undigested non-product / uneven license / compression doctrine teeth). Commit `4dd7b9b`.

**G3 — sandbox EPERM misread as capability boundary (bc0c).** On the ASCC mount I hit `Operation not permitted` from the sandbox and initially reported "can't do this". The correct layer was always available: Cowork exposes Windows-MCP + desktop-commander MCPs that run commands directly on the host. The sandbox `EPERM` is a *sandbox* fact, not a capability fact. Fix was behavioral, not code: always probe `sandbox → host-MCP → user` in that order before degrading.

**G4 — multi-project mount ≠ multi-project focus (381c).** With ASCC mounted alongside Myco, I almost wrote hermes-agent/mempalace forage `--why` text as "for ASCC's Hard Contract…". The mount was material for Wave 9 upstream dogfood, not a shift in focus. Current-session focus SSoT is `MYCO.md` hot-zone "当前阶段", not "what directories are mounted". Fix was behavioral; optional future hardening: add a "session focus = ?" step to Myco session boot.

**G5 — craft conclusion wasn't committed until it was eaten (f153).** Wave 9 craft reached ACTIVE @ 0.91 inside my working memory before any note or lint could see it. The conclusion note itself closed the loop: it's the trace that makes the kernel-contract decision attributable and auditable.

## Meta-pattern: new-substrate happy-path-only validation

All five events trace to the same anti-pattern. When a new substrate goes live (upstream absorb, forage, payload-hygiene policy, multi-mount sessions), the pre-launch validation was happy-path unit tests plus one spec-shaped demo. The moment a real asymmetric input arrived (directory instead of file; `.md` article with non-permissive license; concurrent mounts of unrelated projects; bundle that spans sessions instead of living in one), a latent gap surfaced and got fixed in place.

Rough hit count: **5 out of 5 Wave 9 surprises fit this shape**. None were implementation bugs in the narrow sense — each was a spec/impl gap where the spec covered the intended use but not the obvious neighboring use. This is the same shape as classic "tested the thing we built, not the thing that'll actually arrive".

### What the pattern implies for Gear 4 (not yet promoted)

Discipline from `evolution_engine.md` says craft-promote to Gear 4 only after the **sixth** occurrence. Wave 9 is the fifth dogfood-surfaced new-substrate gap in the Myco timeline (counting L1 backtick, `--entry-point` discoverability, L14 subtree, article gitignore, focus drift as five; the Wave 6–8 ones are older but match shape). Holding off promotion. When it fires a sixth time — most likely on the next new substrate — this retrospective becomes the seed for a craft doc proposing a pre-launch "adversarial-input dry run" step: before shipping any new substrate, run it against one deliberately-ugly input (directory manifest, proprietary license, cross-session bundle) and update spec before CLI lands.

### What the pattern already changed (Wave 9-local)

- Dual-path lint (`myco lint --project-dir .` + `scripts/lint_knowledge.py` shim) is now the committed gate for "15/15 green".
- Payload-off-by-default became the stated Wave 9 principle in `forage/README.md` — any future substrate defaults to "new layer never leaves the local machine" unless opt-in.
- Sandbox bypass probe order (`sandbox → host MCP → user`) is now in working-memory convention, will get re-captured if it slips again.
- `upstream_inbox_pressure` and `forage_backlog` are both first-class hunger indicators; the three channels (inbound forage, internal notes, outbound upstream) are now all visible in `myco hunger`.
