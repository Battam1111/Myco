# Wave 61 经验：并行 agent 瘦身方法论

**核心经验**：对大面积、低耦合的瘦身/修复任务，5 opus agents 并行 + 各自 ≥30 tool uses 强制令（Rule 2），总 304 tool calls，效率远超串行。关键设计：(1) 按文件/子系统切分 agent 边界，避免并发编辑同一文件；(2) 每个 agent 的 prompt 必须含're-read the audit before editing'，因为上游 audit agent 的 finding 有时过期/错位；(3) 每个 agent 负责自己的 syntax-check + 局部 pytest，不依赖 orchestrator 事后兜底；(4) 父 agent 只做最终 full regression + substrate 同步。**边界经验**：识别'不可无损合并'的分歧——_load_canon 4 份实现对应 4 种错误处理契约（silent / dict-return / propagate / sys.exit），合并会抹掉 CLI/MCP/graceful 模式差异，属 A/B 判准中'删除会丢能力'的 A 层成分即使形式上在 B 层（代码）。**反模式教训**：audit agent 报的 LoC 估算偏乐观，实际 Δ=-53 而非 -250~-400；thin test 删除比代码合并收益更大（16 tests vs 57 LoC）。**流程可复用**：未来大 refactor → audit 1 轮并行 → 综合成 HIGH/MED/LOW 清单 → HIGH 再并行执行 → 父节点只做 regression。
