# A/B 二元结构：Myco 瘦身的判断边界

Myco 是复合结构，瘦身前必须先分层：

## A. 知识层（substrate / 内容物）— 不瘦身

- `notes/` 里的 `n_*.md`（消化过的知识单元）
- `_canon.yaml` 里的事实条目（数字、名称、路径、决策）
- `log.md` 时间线
- mycelium graph 的 backlink 连接
- `MYCO.md` 里的 identity anchors

这些是代谢产物，是 Myco 存在的目的。随着使用只会越积越多，瘦身它们等于自我截肢。

## B. 工程层（infrastructure / 代谢器官）— 瘦身对象

- `src/myco/` 下的 Python 代码（CLI、MCP server、immune、notes io、condense 等）
- 29 维 immune lint（L0-L28）—— 多少是冗余/重叠/低价值？
- 25 个 MCP 工具 —— 多少是语义重复的（eat/absorb/ingest/scent/forage 这组值得审）？
- hooks、skills、templates —— 多少是示范性冗余？
- `tests/` —— 421 个测试里多少过拟合实现细节？
- `docs/` 下的协议/辩论/叙事 —— 多少应该 compile 到 wiki 后归档？

**瘦身 B 的标准**：单位价值的代码行 / 工具 / 维度。能不能用更少的机制表达同样的代谢契约？

## 试金石（判断边界）

- 删掉它之后，**已有的知识会丢吗**？→ 会丢 → 属于 A，不动
- 删掉它之后，**只是某种写入/检查/捕获的便利性下降**？→ 属于 B，候选瘦身对象

## 实操推论

- 删 immune 的某维 → B（便利性下降，但已有 notes 不会消失）
- 合并 eat / absorb / ingest → B（语义收敛，存量 notes 保留）
- 删某篇 `notes/n_*.md` → A（直接知识丢失，除非先 condense 并保留 back-reference）
- 删某个 docs/current/ 辩论 → 先 compile 到 wiki 再归档（A 中的结论保留，B 中的过程存档）
