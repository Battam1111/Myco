## 调研总结（传统手艺前置素材）

### Hermes-Agent 核心发现

| 维度 | Hermes 方案 | 与我们系统的关系 |
|------|-----------|---------------|
| **记忆架构** | 四层：Prompt Memory（MEMORY.md/USER.md, 3575 char limit）→ Session Search（SQLite+FTS5）→ Skills（~/.hermes/skills/）→ Honcho User Model | 我们的四层（L1→L1.5→L2→L0）是独立发明的同构设计，但我们有 _canon.yaml SSoT 和 Lint 机制，Hermes 无 |
| **技能系统** | SKILL.md + YAML frontmatter，自声明式（平台、配置、依赖）。26 个类别。agentskills.io 开放标准 | 我们没有"技能"概念——wiki 页面是静态知识，不是可执行的流程。这是 Hermes 领先之处 |
| **自改进循环** | 周期性 nudge（无用户输入时触发自检）→ 技能创建（≥5 tool calls / 错误恢复 / 用户纠正 → 自动写技能）→ 技能 patch（执行中发现更好方法→ 更新技能） | 我们的 Gear 1-4 是类似机制但更结构化；Hermes 的 nudge 是自动化的，我们的是手动触发 |
| **技能生命周期** | create → use → patch（优先于 edit）→ delete。patch 只修改差异，保护已有内容 | 我们的 wiki 页面没有明确的生命周期管理 |
| **上下文管理** | 压缩时保留"谱系链"（lineage），可追溯压缩前原始对话 | 我们靠 L0 Archive JSONL 实现类似功能，但更粗粒度 |
| **部署** | 6 种终端后端 + 单一 gateway 抽象多平台 | 不适用于我们（单用户研究项目） |

**Hermes 最值得借鉴的 3 个机制**：
1. **Periodic nudge**：Agent 主动回顾，不依赖人工触发。我们的 Gear 2 是手动的。
2. **技能 = 可执行过程知识**：不是"知道什么"而是"怎么做"。我们的 wiki 是陈述性知识，缺乏过程性知识。
3. **Patch > Edit**：渐进修改优于全量重写，保护已有内容。我们的文档编辑没有这个原则。

### Nuwa-Skill 核心发现

| 维度 | Nuwa 方案 | 与我们系统的关系 |
|------|---------|---------------|
| **认知蒸馏** | 8 部分 SKILL.md：身份 → 心智模型(3-7) → 决策启发式(5-10) → 表达 DNA → 时间线 → 价值观 → 反模式 → 诚实边界 | 我们的知识编译（wiki 页面）只有粗略的模板；Nuwa 的 8 部分结构远更精细 |
| **六路并行研究** | 著作 → 对话 → 表达片段 → 外部视角 → 决策记录 → 时间线。优先级：一手材料 > 长对话 > 社交媒体 > 第三方评论 | 我们的传统手艺是"辩论式"的，Nuwa 是"采集式"的。互补关系 |
| **三重验证** | 跨领域复现（≥2域）+ 生成力（能预测新立场）+ 排他性（非所有聪明人共识） | 我们没有验证"知识是否具有排他性"的机制——很多 wiki 内容可能是 generic wisdom |
| **矛盾保留** | 矛盾分类为：时间性（演化中）/ 领域性（上下文依赖）/ 结构性（价值张力）——不试图消解 | 我们的辩论记录保留了分歧，但没有对矛盾进行分类 |
| **Agentic Protocol** | 技能在遇到事实性问题时必须调用搜索工具，不靠插值 | 与我们的 W4 在线验证同构 |

**Nuwa 最值得借鉴的 3 个机制**：
1. **排他性作为质量指标**：如果所有人都知道，就不值得蒸馏。我们的 wiki 可能包含太多"常识性"知识。
2. **矛盾分类而非消解**：保留张力，分类其性质。适用于我们的 evolution_engine。
3. **结构化蒸馏模板**：8 部分格式远比我们的 wiki 模板精细。wiki 页面应有更清晰的内部结构标准。

## 传统手艺 Round 1：两个项目对我们系统的启示综合

### 🔴 攻方（激进派）：我们系统需要根本性重构

**论点**：我们的系统本质上是一个**文档管理系统**（Documentation Management System），而 Hermes-Agent 是一个**知识操作系统**（Knowledge Operating System）。差距不在细节，而在范式。

**证据链**：
1. 我们没有"技能"概念。wiki/algorithms.md 告诉 Agent "12 个算法变体是什么"，但不告诉它"如何部署一个新算法到 HPC"——这个过程知识每次都要重新发明（见 system_description_v2 弱点 C 临时脚本黑洞）。
2. 我们的进化机制是手动的。Gear 1-4 需要 Agent 主动记住执行。Hermes 的 nudge 是自动触发的。
3. 我们的 Lint 检查文档一致性，但不检查知识的**实用性**——一个 wiki 页面即使格式完美、引用一致，但内容如果是 generic wisdom，Lint 也会 PASS。
4. C:\temp 脚本黑洞说明：我们的系统只能留存陈述性知识，不能留存过程性知识。

**结论**：需要引入"技能层"——可执行的过程知识，与 wiki（陈述性知识）并列。

### 🔵 守方（保守派）：渐进改进优于范式颠覆

**论点**：我们的系统已经在 7 天内从 949 行单文件进化为 17,000+ 行四层架构——它的核心设计（索引/编译/原始资料分离 + SSoT Lint + 近端丰富化）是 Hermes 没有的独特优势。照搬 Hermes 是倒退。

**反驳攻方**：
1. "技能层"本质上就是我们的 operational_narratives.md（N 系列）+ scripts/。N-001 "HPC Large File Deploy" 就是一个命名过程。缺的不是范式，是**覆盖率**——只有 3 个叙事，应该有 10+。
2. Hermes 的 nudge 机制听起来好，但实际效果取决于 Agent 能力。我们通过 WORKFLOW.md 会话规程（§5）明确了"会话结束必须做什么"，这比隐式 nudge 更可靠、更可审计。
3. Lint 的价值不在于检查知识实用性（那是人类判断），而在于防止文档腐化。Hermes 没有 Lint，它的 MEMORY.md 3575 字符限制是用**丢弃**来解决腐化，我们用**检查**来解决。
4. C:\temp 脚本黑洞的解决方案不需要新范式——只需要一个脚本归档协议（有价值的临时脚本 → scripts/ 目录 + CLAUDE.md 索引）。

**结论**：保持四层架构不变，在现有框架内吸收 Hermes 和 Nuwa 的具体机制。

### 🟢 综合（裁判视角）

**判定**：守方在架构层面正确（四层架构不需要推翻），但攻方在"过程性知识缺失"这一判断上也是正确的。

**综合方案**：
1. **不引入独立的"技能层"**——过程性知识通过**扩展 operational_narratives.md 的覆盖率和结构化程度**来实现（从 3 个增长到 10+，参考 Hermes 的 SKILL.md 格式化方法）
2. **引入"排他性检查"到 Lint**——新增维度检查 wiki 页面是否包含项目特有的操作知识，而非 generic wisdom
3. **引入"矛盾分类"到 evolution_engine**——活跃张力（W9）目前只是列出张力，应该分类为时间性/领域性/结构性
4. **增强 Gear 2 的自动性**——虽然不能像 Hermes 那样做周期 nudge（我们不是持续运行的），但可以在会话规程中增加"自动触发 Gear 2"的条件

**置信度**：85%

## 传统手艺 Round 2：系统命名辩论

### 命名需求分析

**约束条件**：
- 中英文均可使用（开源面向国际社区）
- 不与 ASCC（论文名）混淆——系统名 ≠ 项目名
- 应反映系统本质（LLM Agent 跨会话知识管理 + 自进化）
- 便于 GitHub 仓库名（短、有辨识度、可搜索）
- 不与已有知名项目重名

**系统本质的一句话描述**：一套让 LLM Agent 在有限上下文窗口和多次会话间保持连续认知、积累知识、自我进化的框架。

### 候选名单（第一轮 8 个）

| # | 名字 | 含义 | 优势 | 劣势 |
|---|------|------|------|------|
| 1 | **Mnemos** | 希腊语 Μνήμη（记忆女神 Mnemosyne 词根），= memory | 雅致、短、国际化、无重名 | 可能被误读；太偏"记忆"，系统不只是记忆 |
| 2 | **Anamnesis** | 柏拉图"回忆说"——知识是灵魂对真理的"回忆" | 哲学深度与系统理念高度契合（Agent 每次新会话"回忆"前世知识）| 太长、难拼、太学术 |
| 3 | **Loom** | 织布机——将碎片知识编织为连续认知 | 短、好记、意象好 | GitHub 上已有 loom 项目多个 |
| 4 | **Engram** | 神经科学"记忆痕迹"——记忆在神经系统中的物理编码 | 精确描述系统功能；有科学背景 | 生僻词；GitHub 有 engram 项目 |
| 5 | **Threadweave** | 线 + 编织——跨会话的知识线索编织 | 意象清晰 | 太长 |

| 7 | **Palimpsest** | 羊皮纸被刮去重写但底层文字仍可辨——知识层叠 | 完美隐喻（新会话覆盖旧会话但底层知识保留）| 太长、太生僻 |
| 8 | **Ariadne** | 给忒修斯线团穿越迷宫的女子——知识作为"线索" | 故事好、辨识度高 | 可能与现有项目冲突；只强调"导航"不强调"进化" |

### 决赛辩论

#### 🔴 为 Ariadne 辩护

**核心论点**：Ariadne 的叙事力无可比拟。"Agent 每次新会话进入迷宫（项目复杂性），Ariadne 提供线团（知识系统）让它穿越迷宫不迷路"——这个隐喻一句话就能让任何人理解系统的价值。

**附加优势**：
- 发音国际统一（ah-ree-AD-nee），中文可译为"阿里阿德涅"或直接用英文
- 品牌延展性好：线团（Thread）可以成为子概念——"knowledge threads"、"thread weaving"
- 文化深度：希腊神话在全球技术社区认知度高

**劣势承认**：只强调"导航/引导"，不强调"进化/自改进"。但这可以通过 tagline 补充："Ariadne — Self-Evolving Knowledge for LLM Agents"

#### 🔴 为 Seshat 辩护

**核心论点**：Seshat 是"记录与度量"的女神——她不只记录，还度量（对应我们的 Lint 和 _canon.yaml）。这比 Ariadne（只导航）更全面。

**附加优势**：
- 绝对无重名——GitHub 搜索 "seshat" 只有很少的不相关项目
- 埃及文化在技术社区的新鲜感（不是又一个希腊名字）
- 自然延伸：Seshat's Scroll（文档）、Seshat's Thread（知识线索）

**劣势承认**：发音不直觉（SEH-shaht），中文更难——"塞莎特"？听起来不优雅。

#### 🔴 为 Mnemos 辩护

**核心论点**：最短、最好拼、语义最直接（记忆）。开源项目名不需要深刻叙事——需要的是**好记、好搜、好引用**。

**劣势承认**：语义只覆盖"记忆"维度，不覆盖"进化"维度。

### 裁判裁决

**初选：Ariadne**（75% 置信度，基于当时对系统的理解）

**⚠️ 后续推翻**：在深入头脑风暴确定系统愿景后（见 `vision_debate_2026-04-08.md`），Ariadne 因"格局太小"（只抓"导航"，未抓"活的/代谢/元进化"）被放弃。第二轮命名辩论（8→5→3）最终选定 **Myco**。

**最终选择：Myco**（来自 mycelium 菌丝网络）。详见 `myco_vision_2026-04-08.md`。

## 传统手艺 Round 3：系统进化方案

### 从 Hermes 和 Nuwa 借鉴的具体改进项

基于 Round 1 的综合结论，以下是按优先级排序的具体改进：

### Tier 1 — 即刻执行（不需辩论，ROI 最高）

**E1. 扩展 Operational Narratives 为 "Procedures"（过程知识层）**

从 Hermes 的技能系统借鉴，但不照搬。我们不需要 YAML frontmatter 和自动发现——我们需要的是**有命名的可执行过程**。

当前 N-001~N-003 重命名为 **P-001~P-003**（P = Procedure），新增：
- **P-004**：新容器/HPC 环境验证流程（OSMesa + 依赖 + smoke test）
- **P-005**：desktop-commander 长任务执行模式（后台执行 + 文件读取）
- **P-006**：大规模审计流程（结果目录 + 日志 + 代码完整性 + 系统卫生）
- **P-007**：P4/P5 调度器启动流程（kill旧 → rotate log → 配置参数 → tmux launch → 验证）

格式升级（借鉴 Hermes SKILL.md）：
```markdown
## P-NNN: [过程名称]

**触发条件**：[什么时候应该执行这个过程]
**前置检查**：[执行前需要确认的事项]
**步骤**：
1. ...
2. ...
**已知陷阱**：[历史踩坑记录，带日期]
**最后验证**：[执行完成后如何确认成功]
**历史执行**：[过去成功/失败的执行记录]
```

**E2. 修复 SESSION_PROTOCOL.md 的 deprecated 引用**

将 `docs/server-workflow.md` 引用替换为 `wiki/ssh_hpc.md`。

**E3. 清理 _canon.yaml 进度字段**

删除 `progress` 节（p4_completed, p4_estimated_completion 等），在注释中说明进度跟踪责任在 CLAUDE.md 热区。

**E4. 写 N-004 和 N-005 操作叙事**（实际上是 P-004 和 P-005，按 E1 新格式）

### Tier 2 — 需要设计但不需辩论

**E5. Wiki 页面内部结构标准化（借鉴 Nuwa 8 部分模板）**

当前 W8 Wiki 模板只规定了 header/footer，没有规定内容内部结构。借鉴 Nuwa 的精细化方法：

不同类型的 wiki 页面应有不同的内部结构模板：
- **实体页**（algorithms, environments）：定义 → 枚举 → API → 已知问题
- **概念页**（theoretical_framework, paper_strategy）：核心命题 → 证据链 → 反论 → 应用
- **操作页**（ssh_hpc, known_bugs）：快速参考 → 详细步骤 → 陷阱记录 → 预防清单
- **分析页**（experiment_design, results）：Claims → 数据 → 验证状态 → 开放问题

**E6. 临时脚本归档协议**

从 Hermes 的技能创建触发条件借鉴（≥5 tool calls / 错误恢复 / 非显然流程 → 自动写技能）：

**归档触发条件**：
- 临时脚本在 ≥2 个会话中被重新发明 → 必须归档到 scripts/
- 临时脚本涉及 ≥3 个步骤的复合操作 → 应该归档
- 审计类/检查类脚本 → 始终归档（因为审计是反复需要的）

**归档流程**：
1. 从 C:\temp 复制到 scripts/（必要时重命名、加注释）
2. 更新 CLAUDE.md §4 脚本索引
3. 在 log.md 追加 `script` 类型条目

**E7. docs/current/ 生命周期标签**

不删除辩论文档（它们是不可变的历史），但添加**前缀状态标签**到 CLAUDE.md 索引：
- `[ACTIVE]` — 仍在使用中的辩论/设计
- `[COMPILED]` — 结论已编译到 wiki，原始记录仅供溯源
- `[SUPERSEDED]` — 被更新版本取代

### Tier 3 — 路线图项目（需要更多思考）

**E8. 知识排他性检查**（借鉴 Nuwa 三重验证）

新增 Lint 维度：检查 wiki 页面中是否存在"所有人都知道的常识"内容。实现方式：维护一个"排除列表"在 _canon.yaml 中，记录哪些内容是项目特有的（排他性知识）vs 通用知识。

**E9. 矛盾分类机制**（借鉴 Nuwa）

在 evolution_engine.md 的 W9 活跃张力中增加分类：
- 时间性张力（会随项目进展自然消解）
- 领域性张力（在不同上下文中有不同正确答案）
- 结构性张力（系统内建的设计取舍，不可消解）

**E10. 自动化 Gear 2**（借鉴 Hermes periodic nudge）

在会话规程中增加"自动触发"条件：
- 会话持续 >2 小时 → 中间触发 Gear 2
- 遇到 ≥3 个 friction 事件 → 触发 Gear 2
- 上下文压缩前 → 强制触发 Gear 2

## 传统手艺 Round 4：交叉检验 + 风险评估

### 对 Round 1-3 的自我挑战

**挑战 1**："Procedures 过程知识层"是否只是换了个名字的 operational_narratives？

**回应**：不完全是。关键区别在于**结构化程度**。当前 N-001 是一段叙事文字，新的 P-001 格式有明确的"触发条件→前置检查→步骤→陷阱→验证→历史"六部分。这个结构来自 Hermes SKILL.md 的 frontmatter 理念，但适配到我们的非自动化场景。叙事部分保留在"已知陷阱"中（W6 近端丰富化），但不再是全部。

**挑战 2**：Ariadne 这个名字是否太"文艺"，不适合技术社区？

**回应**：先例充足——Kubernetes（希腊语"舵手"）、Prometheus（希腊神话）、Apollo（同）、Hermes Agent（同）。技术社区对希腊/神话命名有很高的接受度。但**如果用户偏好更务实的名字**，Mnemos 是好备选。

**挑战 3**：E5 Wiki 模板化会不会让 wiki 变得僵化？

**回应**：模板是**指导而非强制**。Nuwa 的 8 部分结构是"蒸馏人"用的，我们的模板是"编译知识"用的，目的不同。每种页面类型的模板应该列出"推荐包含"的部分，而不是"必须包含"的部分。

**挑战 4**：从两个项目借鉴这么多机制，会不会让系统变得过于复杂？

**回应**：这是最严肃的风险。当前系统的 W1-W12 已经很多了。如果全部实施 E1-E10，原则数量会更多。

**缓解**：只实施 Tier 1（E1-E4，都是具体修复/补充），Tier 2 按需逐步引入，Tier 3 留作路线图。不在单次会话中全部实施。
