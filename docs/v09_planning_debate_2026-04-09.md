## 背景：v0.x 的现状

**已有内容**（13 个文件）：
- 模板层：`CLAUDE.md`、`WORKFLOW.md`、`_canon.yaml`、`log.md`
- 文档层：`architecture.md`、`evolution_engine.md`、`vision.md`
- 脚本层：`myco_init.py`（三级 bootstrap）、`lint_knowledge.py`（9维检查）
- 示例层：`examples/ascc/README.md`

**v0.x 的本质局限**：整套系统是从 ASCC（学术论文项目）里蒸馏出来的。模板、示例、lint 检查逻辑——所有东西都带着 ASCC 的隐性假设。拿它去 bootstrap 一个"软件产品项目"或"数据分析项目"，会不会跑通？目前不知道，也没有验证过。

## 第一轮：v0.9 的成功定义之争

### 立场 A：最小主义定义
> "v0.9 = myco_init.py 在第二个项目上成功运行，lint 全 PASS"

**理由**：现在已经可以做到（我们测试过）。v0.9 应该专注于让这件事更流畅、更文档化，而不是加新功能。

**反驳**：这个定义太空洞。`myco_init.py` 只是创建文件，lint 只是检查文件格式。一个全 PASS 的空项目对使用者毫无价值。真正的问题是：**初始化后的 Myco 能否帮助一个陌生的 Agent 真正完成工作？**

---

### 立场 B：质量主义定义
> "v0.9 = 一个完全不了解 ASCC 的人，拿到 Myco 后能在 30 分钟内为新项目建立有效的知识系统"

**理由**：这是真实的用户体验目标。技术上能跑通（立场 A）和实际能用（立场 B）之间有巨大鸿沟。

**反驳**：这个目标太宽泛，无法用代码和文件来验证。"有效"怎么定义？30 分钟是主观感受。

---

### 立场 C：行为主义定义（综合 A + B）
> "v0.9 = 一个新项目在 Myco 初始化后，**其首次会话的 Agent** 能仅凭 CLAUDE.md 一个文件，完成：
> 1. 理解项目状态
> 2. 知道去哪找更多信息
> 3. 知道如何记录新的知识
> 4. 知道如何运行 lint 自检"

**理由**：这四个行为是可以客观验证的，不依赖人类的主观感受，也不只是"文件存在"这种低标准。

**暂定结论**：采用立场 C，四个可验证行为作为 v0.9 的验收标准。

---

## 第二轮：现有 v0.x 的真实差距在哪里？

### 诊断：对着立场 C 的四个行为，逐一检查 v0.x

**行为 1：理解项目状态**
- 现状：`templates/CLAUDE.md` 有 `{{PROJECT_NAME}}`、`{{CURRENT_PHASE}}` 等占位符，但**热区内容是空的**。Agent 填完占位符后，热区只有模板说明，没有真实内容。
- **差距：初始化后热区是空的——Agent 启动时什么都不知道。**

**行为 2：知道去哪找信息**
- 现状：CLAUDE.md 里有 wiki/docs 索引，但索引是空的（wiki/ 目录为空）。
- **差距：空索引没有价值，Agent 不知道第一个 wiki 页面应该写什么。**

**行为 3：知道如何记录新知识**
- 现状：WORKFLOW.md 里有 Wiki 模板（4 种类型），但是在一个很长的文档的中间，不容易找到。
- **差距：知识记录路径不明显——需要翻阅 192 行的 WORKFLOW.md 才能找到模板。**

**行为 4：知道如何运行 lint**
- 现状：CLAUDE.md 底部有一行提示，WORKFLOW.md 里有说明。
- **评估：这一条相对 OK，不是主要问题。**

### 诊断小结

核心差距集中在两个地方：
1. **初始化后的"第一步"不清晰**——Agent 拿到一堆空文件，不知道做什么
2. **知识创建路径太长**——从想到"我要记录这个知识"到找到模板，步骤太多

## 第三轮：解决方案之争

### 方案 X：添加"快速启动指南" QSG
在 CLAUDE.md 模板里增加一个「首次会话清单」章节：
```
## 新项目启动清单（首次会话必读）
1. 把 {{CURRENT_PHASE}} 改为你的实际阶段
2. 在热区写入 2-3 行最重要的当前状态
3. 创建第一个 wiki 页面：`wiki/project_overview.md`
4. 运行 `python scripts/lint_knowledge.py` 验证系统健康
```

**优点**：简单、直接，不需要新文件。
**缺点**：是文字说明，Agent 可能忽略或跳过。

---

### 方案 Y：创建 `wiki/project_overview.md` 模板
把 v0.x 里"wiki 页面只有目录"改为"初始化时自动创建一个填好结构的 `project_overview.md`"。

**优点**：Agent 有个可操作的具体起点，不是面对空目录。
**缺点**：增加了一个半空的文件，可能误导 Agent 认为系统已经有内容了。

---

### 方案 Z：创建独立的 `QUICKSTART.md`
独立于 CLAUDE.md 和 WORKFLOW.md，专门放"第一步怎么用 Myco"的 5-10 分钟指南。

**优点**：不污染现有文件，面向人类用户（而非 Agent）。
**缺点**：又多一个文件，增加认知负担。可能和 README 重叠。

---

### 方案组合建议（待讨论）

以上三个方案并不互斥。一个初步的组合可能是：
- **方案 X**（首次会话清单）→ 改进 CLAUDE.md 模板：让 Agent 知道第一步做什么
- **方案 Y**（project_overview 模板）→ 给 Agent 一个具体的起点
- README.md 增加"第一个小时你会经历什么"章节 → 给人类用户
