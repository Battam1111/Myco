# Migration: v0.5.7 → v0.5.8

**Applies to**: any substrate upgrading from Myco v0.5.7 (or earlier
0.5.x) to v0.5.8+.

**Summary**: v0.5.8 promoted R6 ("write only to the allowed
surface") from a lint-time reporter to a verb-time mechanical
enforcer. Every substrate-mutating verb now raises
`WriteSurfaceViolation` (exit 3) when it would write to a path
outside `_canon.yaml::system.write_surface.allowed`. Two operational
exit codes differentiated; fresh-substrate layout slightly expanded.

---

## TL;DR

Most substrates upgrade without code changes. Do this:

1. Run `pip install -U myco` (or `pip install 'myco==0.5.8'`).
2. Restart any long-running MCP servers so they pick up the new kernel.
3. Run `myco immune` once to see if `PA1` flags any uncovered write
   paths. If it does, add a covering pattern to
   `_canon.yaml::system.write_surface.allowed`.
4. Run `myco hunger`. If it reports `contract_drift: true`, run
   `myco assimilate` to sync `synced_contract_version`.
5. You're done. Continue using verbs as before.

If something broke, keep reading.

## Behavioural changes (may affect operators)

### 1. `WriteSurfaceViolation` on write-surface mismatches

**What changed**: eight verbs (`eat`, `sporulate`, `digest`,
`assimilate`, `fruit`, `ramify`, `molt`, `boot_brief`) now call
`check_write_allowed(ctx, path, verb=...)` before any `write`.
If the target doesn't match any pattern in
`_canon.yaml::system.write_surface.allowed`, the verb raises
`myco.core.write_surface.WriteSurfaceViolation` (exit 3).

**When this fires at upgrade time**: your `_canon.yaml` was authored
against a narrower write-surface than what verbs need. The v0.5.8
defaults are:

```yaml
system:
  write_surface:
    allowed:
      - "_canon.yaml"
      - "MYCO.md"        # or your entry_point filename
      - "notes/**"
      - "docs/**"
      - "src/**"
      - ".myco/**"       # required for substrate-local plugins
```

**Fix options**:

- **Widen the surface** to cover the v0.5.8 defaults (recommended for
  most substrates).
- **Opt out per-invocation**: `MYCO_ALLOW_UNSAFE_WRITE=1 myco eat ...`.
  The bypass is explicit + auditable. Useful for scripted flows that
  know they're out-of-surface.
- **Keep the violation**: accept that the verb will fail cleanly when
  its target isn't in the declared surface. This is the doctrine-
  intended behaviour.

The `PA1` dimension (new at v0.5.8) surfaces uncovered core paths as
MEDIUM findings so you can see the gap at lint time.

### 2. Exit codes 4 and 5

**What changed**:

- `SubstrateNotFound`: exit code 3 → **4**
- `CanonSchemaError`: exit code 3 → **5**

All other operational failures (including `ContractError` and
`UsageError`) remain at 3. All stay within the `≥3` operational band
the L1 contract reserves.

**What to do**:

- Scripts checking `exit != 0`: no change needed.
- Scripts special-casing `exit == 3` for substrate-not-found or
  canon-schema failures: update to also accept `4` and `5`
  respectively.

### 3. Fresh-substrate directories pre-provisioned

**What changed**: `myco germinate` now creates `notes/raw/` and
`notes/integrated/` up front. Pre-v0.5.8, they were created lazily
by the first `eat` / `assimilate` call.

**What to do**:

- Nothing for existing substrates (the dirs have long been created
  on first use).
- For scripts that assumed `notes/` empty = fresh substrate: switch
  to checking `.myco_state/autoseeded.txt` (the canonical fresh
  marker since v0.4.0).

### 4. LF-only line endings on every kernel write

**What changed**: every `write_text` site in the kernel passes
`newline="\n"` via `atomic_utf8_write`. On Windows this prevents
Python's default CRLF translation from contaminating the
persisted-graph byte hash.

**What to do**:

- Nothing unless you had tooling that required CRLF files. If you
  did, those files were already wrong under `_canon.yaml`
  byte-hash-as-fingerprint semantics — upgrade and audit.
- Committing a `.gitattributes` (v0.5.8 ships one) prevents
  CRLF from ever entering git history going forward.

## Additive changes (no action required)

- 14 new lint dimensions (see CHANGELOG). All default to LOW or
  MEDIUM severity; the default CI gate
  (`--exit-on=mechanical:critical,shipped:critical`) is unaffected.
- 4 new core helpers (`atomic_utf8_write`, `trust.py`,
  `skip_dirs.py`, `write_surface.py`). Importable via
  `from myco.core.io_atomic import atomic_utf8_write`, etc.
- New `check_write_allowed(ctx, path, verb=...)` public API for
  verbs that do their own I/O (e.g. `eat`'s `O_EXCL` loop).
- SE1 dangling-ref check got 2400× faster (set membership vs
  per-edge stat). No caller change needed.
- Forage walker got ~15× faster (skip-dir pruning + short-circuit).
- MP1 BLACKLIST extended with 17 more 2026-era provider SDKs.

## Rollback

If v0.5.8 breaks something in your substrate and you need to step
back:

```bash
pip install 'myco==0.5.7'
```

All v0.5.8 `_canon.yaml` fields are backward-compatible with v0.5.7
(the kernel tolerates unknown fields). Your canon will work
unchanged with the v0.5.7 kernel. The only thing you lose is the
mechanical R6 enforcement — which is exactly what you were trying
to escape, so that's the intended rollback behaviour.

If rollback is surprising / undesirable, file an issue on
[Battam1111/Myco](https://github.com/Battam1111/Myco/issues) with
your `myco immune --json` output from both versions so the v0.5.9+
craft can address the rough edge.
