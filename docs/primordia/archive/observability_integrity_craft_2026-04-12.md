# Observability Integrity Craft (Wave 21)

## Round 1 — Claim & First Attack

### A1. Claim

Two small holes from panorama #2 share the same underlying
failure mode — **a sensor that can report an outdated truth with no
warning**. NH-3 is Wave 17's boot brief file going stale without the
agent knowing; NH-7 is the `myco view` verb producing output the
agent has no reason to trust or use. Both reduce observability
without ever raising an alarm. The fix is to make each sensor either
honest-about-its-state or delete its seat at the table.

### A2. Proposal

1. **New lint dimension L16 — Boot Brief Freshness.** `myco lint`
   compares `mtime(.myco_state/boot_brief.md)` against
   `max(mtime(_canon.yaml), mtime(docs/contract_changelog.md))`.
   If the brief is older, fires HIGH: "boot brief is stale vs
   canon/changelog. Run `myco hunger` to regenerate." Respects
   `system.boot_brief.enabled` — if the injector is off, L16
   returns PASS (no brief to keep fresh). If the brief file simply
   doesn't exist, HIGH: "boot brief missing, run `myco hunger`".
2. **Augment `myco view` for agent use.** Add three new modes on
   top of the existing `<id>` positional:
   - `myco view --next-raw` → print the oldest raw note (oldest
     unprocessed item in the digest queue; zero-argument target for
     `myco digest`).
   - `myco view --tag <tag>` → filter notes whose frontmatter
     `tags` contains the given tag, newest first. Useful for
     "show me everything tagged `friction-phase2`".
   - `myco view --last <N>` → the N most recently touched notes.
   Keep the existing single-id mode working unchanged.
3. **Canon bump** kernel + template v0.19.0 → v0.20.0. No new
   mandatory canon fields; L16 uses existing `boot_brief` block.

### A3. First attacks

- **R1.1** "L16 is noise: every edit to `_canon.yaml` will burn a
  HIGH, training the agent to ignore HIGH."
- **R1.2** "`myco view --next-raw` duplicates `myco hunger`'s
  raw-backlog signal; why another surface?"
- **R1.3** "`myco view --tag` is just `grep -l tag: notes/*.md`.
  Building it into the kernel is feature creep."
- **R1.4** "L16 will fire on any repo where the brief was
  generated by an older hunger run — that's literally every
  existing downstream instance today. Wave 21 will land broken."

### A4. Defenses

- **R1.1 defense (valid, adjust).** The concern is real: canon
  edits are frequent, so L16-HIGH is too noisy. Downgrade to
  MEDIUM. The pathology is still "stale observability", not
  data loss — MEDIUM is the correct severity per the W1/W5
  severity asymmetry doctrine. MEDIUM shows in `myco lint`
  output but does not block `lint.main()` exit 0. Agents see
  it, can choose to run `myco hunger` to clear, but it doesn't
  blanket-block progress.
- **R1.2 defense (partial).** `myco hunger` shows counts (e.g.
  "4 raw notes"). `myco view --next-raw` shows the actual *body*
  of the oldest raw note. Different surface — hunger = dashboard,
  view = contents. Especially valuable during digest loops:
  `myco view --next-raw` → read → `myco digest <id>` → repeat.
- **R1.3 defense.** `grep -l tag: notes/*.md` fails for
  YAML-serialized tags in inline lists vs block lists; also
  doesn't respect status filter. Building it into the kernel
  gives agents a reliable, documented surface that does the right
  thing regardless of YAML style. Still keep it minimal: no
  full-text search, just frontmatter tag match.
- **R1.4 defense (valid, adjust).** On Wave 21 landing, the brief
  *will* be regenerated as part of Wave 21's own dogfood (running
  `myco hunger` to produce the conclusion note). So the kernel
  self-repo is clean. Downstream instances will fire MEDIUM on
  their next lint — which is correct: their brief genuinely is
  stale, they should rerun hunger. The "this will fire on every
  downstream" is a feature, not a bug: it's exactly NH-3's
  pathology being surfaced.

## Round 2 — Implementation Plan & Second Attack

### B1. L16 lint implementation

New function `lint_boot_brief_freshness(canon, root)` in
`src/myco/lint.py`. Logic:

```python
def lint_boot_brief_freshness(canon, root):
    issues = []
    system = canon.get("system") or {}
    cfg = system.get("boot_brief") or {}
    if not cfg.get("enabled", False):
        return issues  # injector off → nothing to keep fresh
    brief_rel = cfg.get("brief_path", ".myco_state/boot_brief.md")
    brief = root / brief_rel
    canon_path = root / "_canon.yaml"
    changelog = root / "docs" / "contract_changelog.md"

    if not brief.exists():
        issues.append((
            "L16", "MEDIUM", str(brief),
            "boot brief is missing. Run `myco hunger` to regenerate; "
            "the entry-point signal block will show stale or absent data."
        ))
        return issues

    brief_mtime = brief.stat().st_mtime
    ref_mtime = canon_path.stat().st_mtime if canon_path.exists() else 0
    if changelog.exists():
        ref_mtime = max(ref_mtime, changelog.stat().st_mtime)

    if brief_mtime < ref_mtime:
        from datetime import datetime as _dt
        issues.append((
            "L16", "MEDIUM", str(brief),
            f"boot brief is stale: mtime "
            f"{_dt.fromtimestamp(brief_mtime).isoformat()} "
            f"is older than canon/changelog "
            f"{_dt.fromtimestamp(ref_mtime).isoformat()}. "
            f"Run `myco hunger` to regenerate."
        ))
    return issues
```

Register in `main()`'s `checks` list after L15.

### B2. `myco view` expansion

In `src/myco/notes_cmd.py::run_view`, add argparse flags in
`src/myco/cli.py` for `--next-raw`, `--tag`, `--last`. Dispatch
order: `--next-raw` → `--last` → `--tag` → positional id.

`--next-raw`: list_notes with status=raw, sort by `created`
ascending, print the first one's body via `read_note`.

`--tag T`: iterate list_notes, load frontmatter, filter where
`T in meta.get("tags", [])`, sort by `last_touched` desc, print
id + title + status table (not bodies; keep terminal-friendly).

`--last N`: all notes sorted by last_touched desc, first N, same
table format as --tag.

Comment annotation: `# Wave 21: observability integrity — L16 brief freshness + myco view agent surface (closes NH-3/NH-7)`.

### B4. Contract changelog entry

### B5. Second attacks

- **R2.1** "`--tag` breaks Unix philosophy — view should show *one
  thing*, not a filtered list. That's `myco hunger --tag` territory."
- **R2.2** "L16 MEDIUM still costs ~1 line of terminal output
  every lint run on stale instances. 100 downstream instances ×
  100 lint runs = 10k lines of noise."
- **R2.3** "What if the brief file gets touched by `chmod` or
  `git checkout`? mtime bumps without content regeneration.
  False-green."
- **R2.4** "`--next-raw` is opinionated about which raw comes
  first. What if two raws have identical `created` timestamps?"

### B6. Second defenses

- **R2.1 defense (accept split).** Fair. Rename `--tag` and
  `--last` to operate through a new hidden subverb or alternate
  verb? No — the simpler resolution is to let `view` continue
  showing *a view*: when given `--next-raw` or a positional id it
  shows one note (body); when given `--tag` or `--last` it shows
  a table. Both are "read-only lenses on the substrate", just at
  different scales (one note vs several). Document this clearly
  in the function docstring and `docs/agent_protocol.md` §5.
- **R2.2 defense.** 1 line of MEDIUM per run is the correct
  volume: it's a prompt for action, not a wall of text. And the
  lint_main return code still distinguishes "all-green" (0) from
  "medium/low only" (0) from "high" (1) from "critical" (2) — no
  CI pipeline breaks. The noise cost is bounded by the
  regeneration cost: running `myco hunger` once clears it.
- **R2.3 defense.** `chmod` doesn't change mtime (only ctime).
  `git checkout` does set mtime to checkout-time, which is fresh,
  so false-green only in the narrow window "brief was written
  before a canon change, then the whole repo was re-checked-out,
  making brief newer than canon". That's a transient edge case,
  not a systemic risk. Accepted.
- **R2.4 defense.** The note id includes a 4-char random hex
  suffix; collisions on `created` produce deterministic tie-break
  by filename (string sort). Document in the `--next-raw`
  docstring. Edge case, not a failure mode.

## 6. Known Limitations

- **L-1** L16 does not check brief *content* — only file mtime.
  A hand-edited brief with fresh mtime but wrong contents will
  pass. Acceptable: hand-editing the brief violates W1/L11
  already (brief is a machine output, not a hand-written surface).
- **L-2** `myco view --tag` only matches frontmatter tags, not
  body text. Full-text search is an explicit non-goal.
- **L-3** `--next-raw` doesn't offer `--next-extracted` /
  `--next-integrated` etc. If future workflows want them, add
  more verbs. For now the only pressing digest-queue surface is
  raw→digesting.
- **L-4** The brief is still regenerated only on `myco hunger`,
  not automatically on canon edit. Auto-regeneration would
  require a file watcher — deferred as out-of-scope for kernel.
