## 生きた基質

現代の LLM エージェントは優秀だが健忘症だ。毎セッションが CPU の冷却起動 —— 生の計算能力のみ、永続性ゼロ。昨日学んだことはセッション境界で蒸発し、チームが先月合意したことは誰も気づかぬ片隅で矛盾に腐っていく。エージェントが立つ「地面」は地面ですらなく、現実に追いつくことを祈る静的ファイルの山にすぎない。

Myco はその地面だ。

> **Myco は AI エージェントのための自律認知基質（Autonomous Cognitive Substrate / 基質 / Substrate / 基質）だ。あなたのエージェントは CPU：生の計算、永続性ゼロ。Myco はそれ以外のすべて —— メモリ、ファイルシステム、オペレーティングシステム（operating system / OS ）、周辺機器 —— そしてこの OS は自己升級する（自己升级 / 自己升級）。すべての進化は非パラメトリック（non-parametric / 非参数的 / 重みは一切触らない / weights never touched）：ディスク上のテキスト・構造・lint ルール。モデル重みは一切変更されない。だから Myco はエージェントベンダーをまたいで動き、モデル交換を生き延び、LLM エージェントが最も得意とする媒体の中で価値を蓄積し続けられる。**

これはメタファを建築のふりをさせているのではなく、それ自体が建築だ。**カーネル（kernel / カーネル / 内核）**はプロジェクト非依存の認知 OS である。各プロジェクトディレクトリは**インスタンス（instance / インスタンス / 实例）** —— その OS 上で動く「アプリケーション」である。カーネルのアップグレードは下流に伝播し、インスタンス内で発見された摩擦は上流に戻る。基質は熱力学的な意味で生きている：入ってくる知識を**代謝（metabolize / 代謝 / 消化系）**し、エージェントの注意力が実際に使える形に圧縮し、もはや場所を稼がないものを**排出（excrete / 排出 / 淘汰）**する。代謝を止めた基質は安定した知識ベースではない —— 死体である。**停滞（Stagnation / 停滞）は死。 永続的進化（Perpetual / 永続）** は nice-to-have ではなく、基質が基質であり続けるための唯一の条件である。

## なぜ Myco か

AI メモリと知識システムの領域は三つのカテゴリに分かれる。Myco は独自のカテゴリにいる。

| | メモリ・アズ・ア・サービス | 知識コンパイル | 知識有機体 |
|---|---|---|---|
| **代表例** | Mem0、Zep、LangMem | Karpathy LLM Wiki | **Myco** |
| **中核動詞** | 保存と検索 | コンパイルと索引 | **代謝** |
| **知識ライフサイクル** | なし。データが入り、座り、クエリされる。 | 取り込みと lint のみ。ライフサイクル状態なし。 | 全パイプライン：raw -> digesting -> extracted -> integrated -> excreted |
| **自己検証** | なし | 基本的な矛盾/孤立チェック | 23 次元免疫系（L0-L22） |
| **自己進化** | なし | 静的スキーマ | Craft プロトコルが基質自身のルールを進化させる |
| **排出** | 決してしない。ストレージは増え続ける。 | 決してしない。 | 能動的：死知識検出、自動剪定、時間期限による排出 |

**メモリサービスは保存する。知識コンパイラはコンパイルする。Myco は代謝する。** 取り込むばかりで排出しないシステムは生きた知識ベースではない —— 腫瘍である。Myco は知識に完全なライフサイクルがあるものとして扱う唯一のシステムだ：誕生、消化、検証、圧縮、そして —— もはや場所を稼がなくなったとき —— 排出。生物学の語彙は装飾ではない。それがアーキテクチャそのものだ。

## 五つの核心能力

Myco はメモリレイヤではなく、エージェントランタイムでもなく、スキルフレームワークでもない。その三者の下 —— それらが走る生きた地面の層である。この地面が何をするかを、五つの能力が定義する。

### 1. 知識代謝 —— 七段（seven-step / 七步 / 七段）パイプライン

メモリレイヤは保存と検索をする。基質は**代謝（metabolism / 代謝）**する。入ってくるすべてのコンテンツ —— 読んだ論文、エージェントが当たった摩擦、チャット中に下した設計判断 —— は七段パイプラインを通る：

**Discover → Evaluate → Extract → Integrate → Compress → Verify → Excrete（排出 / 淘汰）**

Discover は入口チャネルをスキャンする。Evaluate はそのコンテンツが基質の注意力を割くに値するかを判断する。Extract は荷重構造を引き抜く。Integrate はそれを既存の知識体に接続する。Compress は注意力を稼がなくなったものを切り落とす。Verify は圧縮が荷重の主張を壊していないか検証する。**Excrete（排出 / 淘汰）** —— 多くの知識システムが忘れるその一段 —— は、腐敗したもの・置き換えられたもの・もはや真でなくなったものを積極的に系から追い出す。**出口のない消化管は腫瘍だ。** 七段パイプラインは Myco の中核的な代謝行為であり、すべての動詞はそれを骨格として命名されている：`eat` / `evaluate` / `extract` / `integrate` / `compress` / `view` / `prune` / `hunger`。

### 2. メタ進化 —— 基質が自分のルールを書き換える

すべての LLM システムは**エージェントの振る舞い**を進化させる。Myco が進化させるのは**エージェントが立つ地面そのもの**である。ある lint ルールが本当の問題をもう捕まえなくなったとき、誤検出を連発し始めたとき、あるいは canon スキーマが自分自身を超えて成長したとき —— Myco の四歯車進化エンジンがそれを察知し、変更を提案する。Hunger Sensing は通常作業中に摩擦を感知し、Session Reflection はセッション終了時に基質自体の改善点を反省し、Milestone Retrospective はマイルストーンで構造的前提に挑戦し、Cross-Project Distillation は終わったプロジェクトから普遍パターンをカーネルへ蒸留し戻す。この四つの内向き歯車に加えて、第五の面 —— **メタボリック入口（metabolic inlet）** —— は外向きで、外部の論文・リポジトリ・記事を引き込む。進化のメカニズムは**変異と選択（mutation / 変異 & selection / 選択圧 / 選择压力 / 选择压力）**：システムが候補変更を生成し、人間はどの変異を生き残らせるかを決める**選択圧**である。

### 3. 自己モデル（Self-Model / 四层自我 / 库存 / Inventory / decay / 退化） —— 自分自身を知る基質

Myco は四層の自己モデルを維持し、「今何を持っているか、それはまだ生きているか」にいつでも答えられる：

- **A · 在庫（Inventory / 库存）** —— 今基質の中に何があるか（自動化）。
- **B · ギャップ感知** —— 基質が**持つべきなのに**まだ持っていないもの（半自動化）。
- **C · 劣化感知（decay / 退化）** —— かつて真だったが、もはや真でないもの。*事実的劣化*（バージョンドリフト、ファイル名変更）と*構造的劣化*（3 日目には正しかったアーキテクチャが 30 日目には間違っている）に分かれる。事実的劣化は `myco lint` がすでに捕まえる。**構造的劣化**は全設計空間で最も硬い未解決問題である。
- **D · 有効性** —— **死知識検出**：基質に入ったあと二度と読まれなかったノート。知識が消費されないなら、それは知識ではなく堆積物である。

この自己モデルこそが、生きた知識システムにとって唯一重要な問いを Myco に投げさせる：*これはまだ自分の場所を稼いでいるか？*

### 4. セッション横断の持続性 —— どの対話よりも長生きする基質

基質の上にあるものはすべて儚い —— 対話は終わり、モデルは世代交代し、ランタイムは再起動する。Myco はそれを超えて残るものだ。次のセッションが**完全な形**で継承する唯一の資産である。どのベンダーのエージェントが次に来ようと関係ない。以前は毎週月曜の朝に再発見されていた暗黙知が、持続的になる。「プロジェクトを把握する」高くつく過程は、たった一度だけ起きる。

### 5. エージェント適応的普遍性 —— 基質がエージェントに合わせて形を変える

基質はエージェントがはめこまれる事前に刻まれた型ではない。Myco のエントリーポイント、圧縮ルール、lint 閾値は、操作している当のエージェント（Claude、GPT、Cursor、Claude Code、まだ見たことのない未来モデル）に合わせて進化することを期待されている。普遍性は正しい方向に走る：**地面が木に適応するのであって、木が地面に適応するのではない。**

## Myco との協働のしかた

伝統的な知識ベースは「人間著者 + 機械読者」を仮定する。Myco はこれを反転させる。**あなたのエージェントこそが基質の主要な主体である** —— エージェントが基質を読み、書き、変更を提案し、基質の上で日々の作業を走らせる。あなたは**時折登場するゲートキーパー（agent-as-subject パラダイム反転）**である：変異を承認し、悪い方向を拒否し、構造的前提が挑戦されるべきときに伝統手芸セッションを招集する、その「**選択圧**」である。

協働を定義する三つの力学：

**変異と選択（mutation / 変異 / selection / 選択圧）。** Myco が変異を担い、あなたが選択を担う。システムは絶えず新しい知識エントリ・新しい lint ルール・新しい圧縮戦略を提案する。あなたの役割はそれらを自分で書くことではなく、どれが生き残るに値するかを判断することだ。継続的にかけられる選択圧だけが、自己進化基質を癌化から守る唯一のメカニズムである。

**生存メカニズムとしての透明性（Transparent / 透明 / 透明）。** Myco のすべての変更は監査可能：すべてのノートに出典があり、すべての lint ルールに辯論記録があり、すべてのカーネルアップグレードに upstream バンドルがある。これは官僚的美徳ではなく、因果の連鎖だ：**透明 → 可読 → 人間の選択圧 → 反癌化（anti-cancer / 癌化 / 癌变）**。透明性を失えば可読性を失い、可読性を失えば選択圧を失い、自己最適化する基質は誰も評価できない方向に **癌化（cancerous / 癌化）** し始める。透明は免疫系であり、書類仕事ではない。

**主体としてのエージェント（agent-as-subject）。** 古典的な「セカンドブレイン（Second Brain）」は人間を運転席に、ツールを助手席に置く。Myco はそれを反転する。エージェントが基質上の主要な読者・書き手・思考者であり、あなたは選択・戦略・構造的判断が必要なときに介入する。この反転こそが、**基質** を「個人知識マネージャ」から区別する根本である。

## 三つの不可変憲法

Myco 内のほぼすべては進化可能だ —— 知識構造、圧縮ルール、lint 次元、進化エンジンそれ自体すら。すべて、以下の三つを除いては。L9 lint 次元が基質アイデンティティのドリフトを防ぎ、L13 lint 次元がすべてのルール変更に監査可能な伝統手芸記録を残すことを強制する。

| # | 憲法 | なぜ荷重になるか |
|---|------|------------------|
| **C1** | **アクセス可能（Accessible）** | どのベンダーのどのエージェントも、エントリーポイントを見つけ、基質を自己説明できなければならない。地面にアクセスできなければ、木は育たない。 |
| **C2** | **透明（Transparent / 透明）** | すべての変更は人間に監査可能でなければならない。これが自己進化するシステムに人間の選択圧を働かせるチャネルである。透明を失う → 選択圧を失う → 基質は **癌化（癌化 / cancerous）** する。 |
| **C3** | **永続的進化（Perpetual / 永続）** | **停滞（Stagnation / 停滞）は死。** 代謝を止めた基質は定義上もはや基質ではない。静的な知識ベースになる —— それこそ Myco が逃れるために作られた失敗モードである。 |

この三つが憲法だ。それ以外はすべて立法である。

## 圧縮は認知である（Compression Is Cognition / 圧縮 / 压缩）

Myco の運用仮定：

> **ストレージは無限、注意力は有限（注意力は有限 / 注意力有限 / attention is not）。**

ディスクは必要に応じて増えるが、エージェントのコンテキストウィンドウは増えない。だから Myco は**決して忘れない** —— コールドストレージから何も削除しない —— が、注意力に流れ込むものは**積極的に圧縮（Compression / 圧縮 / 压缩）**する。圧縮は工学的な配管ではなく、**基質の中核的な認知行為**である。三つの候補基準：

- **使用頻度**：誰にも読まれないページはコールドへ。
- **時間的関連性**：時限的事実は期限で排出。
- **排他性**：エージェントが事前学習ですでに知っていることは基質の場所の無駄。エージェントが持たないものだけを残す。

圧縮は**エージェント適応的**でもある：一つのエージェントに書き下す必要があるものが、より強い事前学習を持つ別のエージェントには冗長かもしれない。基質がエージェントに合わせて調整するのであって、その逆ではない。不可縮のテクスチャは意図的に失われる —— 字面的忠実さが必要なら、生アーカイブを別に保持してほしい。基質が残すのは荷重構造：出典チェーン、決定、なぜそれが真であるかの理由である。

## 五十年の肩の上で

Myco は空から降ってきたわけではない。五つの伝統の肩の上に立っており、それぞれが一つの荷重的洞察を提供している：

- **Karpathy LLM Wiki** —— 構造化された知識コンパイルがエージェントの正しい基質形状であって、チャットログやベクトルストアではない。これが幾何学的前提だ。
- **ポラニー（Polanyi / ポラニー）暗黙知** —— 操作的知性の大部分は暗黙的で、proximal/distal 構造の中に宿り、枚挙では捕まえられない。これが Myco が「事実」だけでなく「手順と物語」を持続化する理由だ。
- **アージリス（Argyris / アージリス）二重ループ学習** —— 単一ループは行動を修正し、二重ループは行動を支配するルールを修正する。これが L-struct / L-meta の分割であり、基質自体が進化しなければならない理由だ。
- **トヨタ PDCA** —— Plan / Do / Check / Act は自己改善システムの基本サイクルである。四歯車はこの PDCA を LLM 基質上にコンパイルした形にほかならない。
- **Voyager スキルライブラリ** —— 接地された実行に基づく反復的スキル蓄積は、何が機能したかを記憶し次のエピソードをその上に築かせれば可能である。これが Cross-Project Distillation 蒸留の形だ。

Myco はこの五つを同じ土台に載せ、一つの代謝として動かす最初のシステムである。詳細は [`docs/theory.md`](docs/theory.md)。

## すでに無意識に走っていた

居心地の悪いが荷重的な真実：**我々はすでに、気づかぬうちにこのシステムの原始バージョンを走らせていた。** 8 日間・80+ ファイルの強化学習研究プロジェクトが四歯車サイクルを完全に手作業で走破した。手動 lint、口頭の摩擦ログ、人間駆動のメタ進化、15+ 回の構造化辯論。機能した。そして次のプロジェクトでも**また**機能した。

<div align="center">

| 80+ ファイル | 10 wiki ページ | 15+ 構造化辯論 | 23/23 lint 次元全緑 |
|:------------:|:--------------:|:--------------:|:-------------------:|

</div>

Myco はすでに野生で自証されたパターンの**形式化**である。v0.x → v1.0 の軌跡は「新しいものを発明する」ことではなく、「手動で動いていたものを自動で動かす」ことだ。Cross-Project Distillation で抽出されたパターンは今カーネルコードベースに住んでいる。我々が頼むのは理論を信じてくれということではなく、80+ ファイルと 15+ 辯論の過酷な研究プロジェクトですでに自証したパターンを信じ、そして「人間がエンジンでなくなる」ための残りのダイヤルを一緒に締めてくれということだ。

## 未解決問題（Open Problems）

Myco はまだ初期だ。最大インパクトの貢献は、以下の六つのうち一つを選んで前に進めることだ：

1. **コールドスタート。** 履歴も canon も摩擦記録もない新規プロジェクトで、基質はどう bootstrap するのか？現在の答：手作りの `myco init` テンプレート。目標：過去の蒸留から基質が自分の bootstrap を学ぶこと。
2. **トリガー信号。** Session Reflection を発火させるのは何か？メタボリック入口を発火させるのは何か？摩擦カウントはプロキシにすぎず、真の信号は未解決研究問題だ。
3. **深層でのアライメント。** Myco が人間にはもう有意義に評価できない深さでルールを進化させた場合、どう整合を保つか？透明は必要だが十分でない —— スケールでも成り立つ**可読な透明性（legible transparency）**が必要だ。
4. **圧縮工学。** 荷重的暗黙知を失わずに、何を、いつ落とすか？三つの候補基準は出発点であって解答ではない。
5. **構造的劣化検出（自己モデル C 層）。** 事実的劣化は捕まえられる。構造的劣化 —— 3 日目には正しかったアーキテクチャが 30 日目には間違っている —— はまだだ。設計空間で最も難しい問題だろう。
6. **死知識追跡（自己モデル D 層）。** 最小可行シードは出荷済み（Wave 18、v1.4.0）—— `myco view` が `view_count` + `last_viewed_at` を書き込み、`myco hunger` が `dead_knowledge` 信号を出力し、`myco prune`（Wave 33）が自動排出経路となる。未解決：view 監査ログ、cross-reference グラフ、基質年齢に応じて進化する適応的閾値。

継続的に維持されているレジストリは [`docs/open_problems.md`](docs/open_problems.md) にある。高インパクトの貢献をしたいなら、一つ選んで突っ込んでほしい。

## 今日から試す

読むのをやめて走らせたいなら：

```bash
# ソースからインストール（PyPI 公開準備中）
pip install git+https://github.com/Battam1111/Myco.git

# カーネル上に新しいインスタンスを立ち上げる
myco init my-project --level 2

# あるいは既存プロジェクトを移行（非破壊；CLAUDE.md はそのまま）
myco migrate ./your-project --entry-point CLAUDE.md
myco lint --project-dir ./your-project     # 基質のベースラインを確立
myco hunger --project-dir ./your-project   # 代謝ダッシュボード
```

**MCP 統合** —— エージェントが自動的に 19 のツールを得る。手動プロンプトは不要：

```bash
pip install 'git+https://github.com/Battam1111/Myco.git#egg=myco[mcp]'
```

リポジトリにはすぐ使える `.mcp.json` が同梱されている。Claude Code、Cursor、または MCP を話すどのクライアントにインストールしても、エージェントは自動発見する：

- **基質の健全性**：`myco_lint` · `myco_status` · `myco_search` · `myco_log` · `myco_reflect`
- **知識代謝**：`myco_eat` · `myco_digest` · `myco_view` · `myco_hunger`

CLI だけでも十分使える。MCP 層は「捕捉」を動作から反射に変える層である。

## 菌糸（Mycelium）

この名前は装飾ではない。菌糸は森の地下にある真菌ネットワークで、それは**配管ではない**。落ち葉を栄養に分解する酵素を分泌する（**代謝**）。効果的な成長経路を記憶し、それに応じて戦略を再配分する（**メタ進化**）。豊富な領域から欠乏領域へ資源を再分配する（**知的圧縮**）。出会うあらゆる樹種の根と共生関係を結ぶ（**エージェント適応的普遍性**）。健全な菌糸こそが、森を「孤独な幹の集まり」ではなく「森」たらしめている理由である。

エージェントは地上の木だ。Myco はその下の、森全体を森にしている生きたネットワークだ。
