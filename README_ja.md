# Myco

**AI プロジェクト向けの知識基質 —— エージェントには見えない矛盾を lint で検出し、プロジェクトの成長に合わせてルール自体を進化させます。**

*Git はあなたのコードを追跡する。Myco はあなたのコードが**何を知っているか**を追跡する。*

## 30 秒デモ

3 つのファイルが 3 つの異なることを言っている。本番で問題になるまで誰も気づかない。

```bash
$ cat _canon.yaml
project:
  current_api_version: "v3"
system:
  stale_patterns: ["v2 endpoint", "api/v2"]

$ cat wiki/api.md
# API Reference
The current v2 endpoint lives at /api/v2/users.

$ myco lint

  L0 Canon self-check          PASS
  L1 Reference integrity       PASS
  L2 Number consistency        ⚠  2 issues
  L3 Stale patterns            PASS
  ...
  L14 Forage hygiene           PASS

  [MEDIUM] L2 | wiki/api.md:3
           Stale pattern 'v2 endpoint': "...The current v2 endpoint lives at..."
  [MEDIUM] L2 | wiki/api.md:3
           Stale pattern 'api/v2': "...lives at /api/v2/users..."
```

2 つの矛盾を捕捉。今後 5 セッション分の静かな複利的誤りを防止。これが基質免疫系の働きです —— しかも上記のルール（`stale_patterns`）自体が `_canon.yaml` のなかでバージョン管理されているので、プロジェクトと一緒に進化していきます。

## クイックスタート

```bash
# ソースからインストール（PyPI 公開は近日中）
pip install git+https://github.com/Battam1111/Myco.git

# 新規プロジェクト
myco init my-project --level 2

# 既存の CLAUDE.md を持つプロジェクト
myco migrate ./your-project --entry-point CLAUDE.md   # 非破壊的
myco lint --project-dir ./your-project                 # ベースラインを確立
```

**MCP 統合** —— エージェントが 9 個のツールを自動取得します（手動プロンプト不要）：

```bash
pip install 'git+https://github.com/Battam1111/Myco.git#egg=myco[mcp]'
```

リポジトリには即使える `.mcp.json` が含まれています。Claude Code、Cursor、あるいは任意の MCP 対応クライアントでインストールすれば、エージェントは自動でこれらを検出します：

- **チェック** · `myco_lint` · `myco_status` · `myco_search` · `myco_log` · `myco_reflect`
- **知識代謝** · `myco_eat` · `myco_digest` · `myco_view` · `myco_hunger`

Myco は `myco` CLI だけでも動きますが、「反射層」（エージェントが会話の流れのなかで知識を自動捕捉する）は MCP を接続したときにのみ存在します。

## Myco ができること

具体的な 3 つの能力を、体感しやすい順に並べます：

### 1. 構文ではなく知識そのものを lint する

Markdown linter は構文をチェックします。Prose linter はスタイルをチェックします。**しかし、wiki の主張が canon ファイルとまだ一致しているかをチェックするものは何もありません。** Myco はそれをやります。`myco lint` は `_canon.yaml`、`wiki/`、`docs/`、`MYCO.md`、そしてあなたが宣言したプロジェクト固有のファイルに対して 15 次元のチェック（L0–L14）を実行し、陳腐化パターン、孤立参照、スキーマ違反、数値の drift が発生した瞬間に捕捉します。

### 2. エージェントに安定した代謝ループを与える

エージェントが論文を読んだ、バグパターンを発見した、設計判断を下した —— その内容を置ける場所がただ一つ存在します：`myco_eat` が raw note として捕獲します。`myco_digest` がそれをライフサイクル（raw → extracted → integrated → excreted）に沿って進め、出所を保持します。`myco_view` であなた（またはエージェント）は tag、status、stage でノートを取り出せます。この基質はセッションをまたぎ、エージェントベンダーをまたぎ、プロジェクト再構築をまたいで生存します。

### 3. プロジェクトの成長に合わせて自身のルールを進化させる

Myco の lint ルールは `_canon.yaml` のなかに住み、構造化された「伝統手工（craft）プロトコル」を通じて進化します（L13 チェックは、ルール変更のたびに監査可能な複数ラウンドの議論記録を残すことを強制します）。**エージェントは新しい lint ルールを提案でき、それらの提案はあなたという選択圧にさらされます。** 通ったルールは基質の一部になります。動かなくなったルールは排出されます。これが L-meta 層です：エージェントを進化させるのではなく、エージェントが立つ地面そのものを進化させるのです。

## 用語集

Myco は生物学的な語彙を使います。以下が普通の英語との対応表です：

| Myco 動詞 | 平易な意味 | CLI | MCP ツール |
|---|---|---|---|
| `eat`（食べる） | コンテンツを耐久性のあるノートとして捕獲 | `myco eat` | `myco_eat` |
| `digest`（消化する） | ノートをライフサイクル（raw → extracted → integrated → excreted）に沿って進める | `myco digest` | `myco_digest` |
| `view`（見る） | status、tag、stage でノートを読む | `myco view` | `myco_view` |
| `lint` | 全ファイルで矛盾と drift をチェック | `myco lint` | `myco_lint` |
| `forage`（採集） | 外部ソース（リポジトリ、論文、記事）を基質に取り込む | `myco forage` | （CLI のみ） |
| `absorb`（吸収） | 下流プロジェクトインスタンスから kernel 改善を同期 | `myco upstream absorb` | （CLI のみ） |
| `distill`（蒸留） | 完了したプロジェクトから普遍的パターンを抽出し将来の kernel 用に残す（Gear 4 の出力） | （ワークフロー、CLI ではない） | — |
| `hunger`（飢え） | 代謝ダッシュボード：raw バックログ、陳腐化ノート、死んだ知識、forage 圧力 | `myco hunger` | `myco_hunger` |

これらの動詞は意図的に比喩的です。Myco の中心的な主張は、知識基質は生きたシステムだ —— 代謝こそが正しいメンタルモデルだ、というものです。この表は、その主張を信じる前にツールを使い始められるよう存在しています。

## 仕組み

Myco はあなたの既存のプロジェクトファイルの横に、4 つの軽量レイヤーを追加します：

```
your-project/
├── MYCO.md           ← エージェントが毎セッション読む（ホットゾーン索引）
├── _canon.yaml       ← 単一の真実の源（lint が照合する正統値）
├── wiki/             ← 構造化された知識ページ（lint 検証済み）
├── docs/             ← 手順、議論記録、進化履歴
├── notes/            ← 消化基質（raw → extracted → integrated）
├── forage/           ← ライセンスゲート付きの外部ソース入口
├── log.md            ← 追記のみのプロジェクトタイムライン
└── src/myco/         ← CLI + MCP サーバー + 15 次元 lint エンジン
```

`myco lint` は**免疫系**です —— 矛盾、孤立ファイル、陳腐化した参照、バージョン drift が複利化する前に捕獲します。15 次元（L0 から L14 まで）は、canon の自己整合性、参照整合性、数値 drift、陳腐化パターン、孤立検出、log カバレッジ、日付整合性、wiki 形式、adapter スキーマ、`.original` 同期、ビジョンアンカー、notes スキーマ、書き込み面の衛生、upstream dotfile 衛生、craft プロトコルスキーマ、そして forage 衛生をカバーします。

### 一行でのアーキテクチャ位置づけ

あなたのエージェントが CPU、Myco が operating system —— そしてこの OS は自己升級します。すべての進化は**非参数的（non-parametric）**です：markdown、YAML、フォルダ構造、lint ルール。モデルの重みは一切触られません。これが、Myco がエージェントベンダーをまたいで一貫して動作し、モデル世代交代を越えて生き残る理由です。

### 三つの不変法則

Myco のなかのすべてのものは進化できます —— 知識構造、圧縮ルール、進化エンジンそのものさえも。ただしこの 3 つを除いて：

1. **アクセス可能（Accessible）** —— 任意のエージェントがエントリーポイントを見つけて基質を自己説明できる。
2. **透明（Transparent）** —— すべての変更は人間によって監査可能。*なぜこれが荷重を担うか*：人間が Myco の選択圧である。透明性が失われれば、選択圧が失われ、選択圧のない自己最適化基質は癌化する。
3. **永続的進化（Perpetually Evolving）** —— 停滞は死。代謝を停止した基質は静的なナレッジベースへと退化する。

### 四つのギア + 代謝入口

Myco の進化エンジンには 2 つの顔があります。4 つのギアは**自律神経系** —— 内部恒常性、すべて v0.x で出荷済み。代謝入口は**消化系** —— 外部吸収、primitive として宣言済み、早くて v1.0 で完成予定。

| ギア | いつ | 何を | 状態 |
|------|------|------|------|
| 1 | 毎セッション | 摩擦を感知 —— 失敗や予期しない挙動を記録 | ✅ |
| 2 | セッション終了時 | 内省 —— 知識システム自体が何を改善すべきか？ | ✅ |
| 3 | マイルストーン | 回顧 —— 構造的仮定に挑戦する | ✅ |
| 4 | プロジェクト終了 | 蒸留 —— 将来のプロジェクトのため普遍的パターンを抽出 | ✅ |
| **代謝入口** | **摩擦シグナル + 周期的巡回** | **外部知識の 発見 → 評価 → 抽出 → 統合 → 圧縮 → 検証 → 排出** | **Primitive 宣言済み；初回の実戦 forage バッチを 2026-04-11 に完了** |

ギア 1〜4 は内向きです。入口は外向きです。消化系のない基質はただのキャッシュです。だからこそ、入口は完全自律形態が後で来るとしても、今宣言しておく必要があるのです。

### 圧縮の教義

Myco の運用前提：

> **ストレージは無限。注意力はそうではない。**

ディスクは要求に応じて成長しますが、エージェントのコンテキストウィンドウはそうではありません。だから Myco は**決して忘れません** —— コールドストレージから何も削除されません —— しかし、注意力に流れ込むものは**積極的に圧縮します**。圧縮は配管ではありません。それは基質の主要な認知行為です。3 つの候補基準：使用頻度（読み出しの少ないページは冷えていく）、時間的関連性（時限付き事実は期限で排出）、排他性（エージェントがすでに持つ常識は基質空間の無駄）。

> ⚠️ *誠実な限界*：圧縮が捕捉するのは、ある知識の荷重を担う決定と出所の連鎖であり、完全な生のコンテキストではありません。還元不可能なテクスチャは失われます。これは Myco が意図的に選ぶトレードです。語彙レベルの忠実性が必要な場合は、別途生のアーカイブを保持してください。

## 類似ツールとの比較

Myco は**メモリ層でもなく**、**エージェントランタイムでもなく**、**スキルフレームワークでもありません**。それらが扱わないレイヤーです。

|  | Myco | Mem0 | Letta / MemGPT | mempalace | Hermes | Claude Code |
|--|---|---|---|---|---|---|
| **カテゴリ** | 知識基質 | メモリ層 | エージェントランタイム | 会話メモリ | エージェントランタイム | エージェント CLI |
| **主要成果物** | `wiki/` + `_canon.yaml` + `notes/` | KV + ベクターストア | 階層メモリ（RAM / disk / archival） | 空間スキーマ（wings/rooms） | セッション DB | `CLAUDE.md` 慣習 |
| **エージェント実行をホストする？** | いいえ —— Claude Code / Cursor の*内部*で動く | いいえ | はい | いいえ | はい | はい |
| **セッション間の契約強制** | ✅ 15 次元セルフ lint | ❌ | ❌ | ❌ | ランタイムキャッシュ不変量のみ | ❌（純粋な慣習） |
| **自己進化ルール** | ✅ Gear 4 + craft プロトコル | ❌ | ❌ | ❌ | ❌ | ❌ |
| **検索ベンチマーク** | —（目的が異なる） | LongMemEval 49% | LoCoMo 74%（GPT-4o mini） | LongMemEval R@5 96.6% 生 | — | N/A |
| **統合方法** | MCP サーバー + CLI | REST API + SDK | Runtime API | MCP サーバー | Runtime | ネイティブ |

**鍵となる洞察**：Myco は*「このプロジェクトの知識はまだ内部で整合しているか？」*という問いに答える唯一の列であり、*「関連するメモリを見つけられるか？」*ではありません。これらは別々の問いで、前者は現状過小にしか対応されていません。

具体的には：
- **Mem0 / Zep / Supermemory** はメモリ層 —— データを保存して取り出します。Myco は**何も保存せず、何も取り出しません**。エージェントがすでに書いているプロジェクトファイルを lint するだけです。
- **Letta / MemGPT / Hermes** はランタイム —— エージェントの実行ループをホストします。Myco は**エージェントをホストしません**。エージェントは Claude Code、Cursor、あるいは任意の MCP クライアントの上で動き、**9 個のツールを通じて Myco を呼び出します**。
- **Claude Code / Cursor / `CLAUDE.md`** は Myco が**その内部で**動く環境です。Myco の `.mcp.json` は既存の Claude Code インストールに lint + 代謝ツールを自動的に追加します。
- **nuwa-skill / agentskills.io / pua** はスキルフレームワーク —— 再利用可能な振る舞いをパッケージ化します。Myco は**スキルフレームワークではありません**。スキルは Myco の上で動き、Myco はそのスキルが操作するプロジェクトを lint します。

上表のベンチマーク数値は、2026 年 4 月時点のベンダー文書、独立系ベンチマークレポート、および Web 検索結果からです。変動する可能性があります。Myco は検索ベンチマークで競争しないことを意図的に選んでいます。目的（検索ではなく検証）が異なり、一対一の比較は誤解を生むためです。

## 未解決の問題

Myco はまだ早期段階です。以下の 6 つの盲点が、貢献する際にもっともレバレッジの高い場所です。継続的に維持されるレジストリは [`docs/open_problems.md`](docs/open_problems.md) にあります。

1. **コールドスタート。** 履歴も canon も摩擦記録もない新規プロジェクトで、Myco はどう自己ブートストラップするか？現状の答え：手作業の `myco init` テンプレート。望ましい形：基質が過去のプロジェクト蒸留から自らのブートストラップを学ぶこと。
2. **トリガーシグナル。** 何が Gear 2 を発火させるのか？何が代謝入口を発火させるのか？摩擦カウントは代理指標にすぎません。正しいシグナルはオープンな研究課題です。
3. **深層でのアライメント。** Myco が人間にはもはや評価できないルールを進化させたとき（深い L-meta）、どうアライメントを保つか？透明性は必要だが十分ではありません —— スケールに耐える*判読可能な*透明性が必要です。
4. **圧縮エンジニアリング。** 何を、いつ、荷重を担う暗黙知を失わずに捨てるか？3 つの候補基準（頻度 / 時間 / 排他性）は出発点であり、解答ではありません。
5. **構造的退化検出（Self-Model C 層）。** `myco lint` は*事実的な*退化（バージョン drift、陳腐化した参照）を捕獲します。まだ*構造的な*退化は捕獲できません —— 3 日目に正しかったアーキテクチャが 30 日目には間違っているような場合です。おそらく設計空間全体で最も難しい問題です。
6. **死んだ知識の追跡（Self-Model D 層）。** D 層は宣言済みだが未実装です。「ノートが 30 日前に基質に入ってから一度も読まれていない」というシグナルはまだありません。最小実行可能シードがロードマップに載っています。

高インパクトの貢献をしたいなら、このなかから 1 つ選んでください。

## 検証

完全な四ギアサイクルを最後まで回した、実在する 8 日間・80 以上のファイルの研究プロジェクトの上に構築されています：

<div align="center">

| 80+ ファイル | 10 wiki ページ | 15+ 構造化議論 | 15/15 lint 次元グリーン |
|:-----------:|:-------------:|:-------------:|:----------------------:|

</div>

より深い主張：**その 8 日間のプロジェクト自体が、Myco の無意識のプロトタイプでした。** 私は基質をデモしていたのではなく、それを*手動で走らせていた*のです —— 人間駆動のメタ進化、手動トリガーの lint、口頭の摩擦ログ。それは機能しました。Myco は、すでに自ら証明されたパターンの形式化です。v0.x → v1.0 の軌道は「新しいものを発明する」ことではなく、「手作業で動いていたものをそれ自身で動かす」ことです。Gear 4 を通じて抽出されたパターンは、現在 Myco kernel コードベースのなかに生きています。[`examples/ascc/`](examples/ascc/) を参照。

Myco は 50 年の系譜の上に立ちます：**Karpathy LLM Wiki**（構造化知識コンパイル） + **ポラニーの暗黙知**（近位 / 遠位構造、運用経験用） + **アージリスの二重ループ学習**（シングルループは行動を直し、ダブルループはルールを直す —— L-struct / L-meta の分離） + **トヨタの PDCA**（Plan / Do / Check / Act を四ギアの底層サイクルとして） + **Voyager のスキルライブラリ**（接地された実行による反復的なスキル蓄積）。[`docs/theory.md`](docs/theory.md) を参照。

## 物語

1 日目、私は 949 行の `CLAUDE.md` を持っていました。すべてが一つのファイルのなか。3 日目までに、同じ指標が 3 つの場所に 3 つの異なる値とともに現れ、私のエージェントは 3 つすべてを自信満々に使いました。しかしより深い問題はその下にありました：新しいセッションのたびに、私のエージェントは同じデプロイスクリプトをゼロから書き直していました —— SSH 設定のルールを忘れたからではありません（それらは文書化されていました）—— どのフラグが重要か、どの順序だと動くのか、何が静かに壊れるのかという*暗黙知*が、セッション境界で消え去っていたからです。知性が失われていたのではありません。知性は**捨てられていた**、繰り返し。

そのとき私は最初の `myco lint` を作りました —— 矛盾を捕獲するためだけではなく、エージェントが根本的に欠いているもの、つまり基質を与えるために。エージェントが「知っている」ことがまだ真であるかを見守り、仮定が腐ったら手を挙げ、古いルールが機能しなくなったら自分のルールを進化させる地面。エージェントが拾い上げるツールではなく、エージェントが立つ床です。

7 日目には、マイルストーンのレトロスペクティブで、摩擦の 40% が「内容を変えたが索引の更新を忘れた」に由来することが判明しました —— そこでシステムは自らのルールを進化させ始めました。8 日目、私はこのパターンがプロジェクト固有ではないと気づきました。私はそれに名前をつけました：**Myco**、*mycelium*（菌糸体）から取って —— あらゆる森の地下にあるあの真菌ネットワークです。菌糸体は配管システムではありません。それは酵素を分泌して落ち葉を栄養に分解します（代謝）。効果的な成長経路を覚え、それに応じて戦略を方向転換します（メタ進化）。豊富な地帯から欠乏する地帯へと資源を再分配します（知的な圧縮）。出会う任意の種の木の根と共生を形成します（エージェント適応的な普遍性）。エージェントは地上の木々です。Myco はその下にある生きたネットワーク、森全体を機能させているものです。
